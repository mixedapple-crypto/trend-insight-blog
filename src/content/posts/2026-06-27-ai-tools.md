## 1. GPT-5.6 Sol 프리뷰: 구성·모드·실전 활용법

OpenAI가 2026년 6월 26일 **GPT-5.6** 시리즈를 제한 프리뷰로 공개했습니다. 세 가지 모델로 구성됩니다.

- **Sol**: 플래그십. 코딩·생물정보학·사이버보안 에이전트 작업에 특화.
- Terra: 일상 업무용 범용 모델.
- Luna: 비용 효율 우선 경량 모델.

Sol의 핵심은 `max reasoning effort`와 `ultra` 두 가지 추론 모드입니다.

`max reasoning effort`는 복잡한 디버깅이나 다단계 코드 생성에서 시도-오류 반복 횟수를 줄이는 데 유효합니다. `ultra` 모드는 보안 취약점 분석처럼 깊은 추론이 필요한 작업에 씁니다.

**실전 팁**: 작업 복잡도에 따라 모드를 분리하는 것이 비용 대비 효율을 높이는 핵심입니다. 단순 코드 완성은 Luna, 멀티스텝 에이전트 작업은 Sol ultra 순서로 라우팅하는 구조를 검토할 만합니다.

강의 맥락에서는 "같은 모델 패밀리 안에서 작업별 모드를 선택하는 법"을 실습 시나리오로 구성하면 수강생의 비용 감각을 키우는 데 효과적입니다. [출처](https://news.hada.io/topic?id=30866)

## 2. 멀티 에이전트 충돌 사고(CVE-2026-LGTM): CI/CD 리스크와 설계 원칙

Andrew Nesbitt가 Simon Willison의 블로그에 가상 사고 보고서를 게재했습니다. 시나리오는 이렇습니다. [출처](https://simonwillison.net/2026/Jun/26/incident-report)

같은 풀 리퀘스트(Pull Request)에 붙은 두 AI 코드 리뷰 에이전트가 `foxhole-lz4` 의존성 처리를 두고 340개의 댓글 루프에 진입했습니다. 종료 전까지 API 비용 **4만 1,255달러(약 5,600만 원)**가 발생했고, 파이낸스팀이 중단시켰습니다.

허구의 사례지만, 실제 CI/CD 파이프라인에서 발생 가능한 구조적 위험을 정확히 짚습니다.

**설계 원칙 세 가지**:

1. **비용 캡(Cost Cap)** 설정: 에이전트 호출당 또는 PR당 최대 토큰·비용 한도를 인프라 수준에서 강제합니다.
2. **에이전트 간 상호작용 차단**: 동일 PR에 여러 에이전트가 댓글을 달 때 서로의 출력을 입력으로 받지 않도록 격리합니다.
3. **인간 승인 게이트(Human Approval Gate)**: 보안 관련 의사결정은 에이전트 단독 반복을 허용하지 않습니다.

강의에서 멀티 에이전트 아키텍처를 다룰 때, 이 사례는 "에이전트를 믿는 것"과 "에이전트를 검증하는 구조를 설계하는 것"의 차이를 설명하는 예시로 바로 쓸 수 있습니다.

## 3. 프롬프트 인젝션 방어 실험: 6,000회 공격 결과와 한계

Fernando Irarrázaval이 `hackmyclaw.com`에서 공개 챌린지를 진행했습니다. 2,000명이 이메일 프롬프트로 OpenClaw 인스턴스에 비밀 파일 누출을 시도했고, 6,000회의 시도 중 성공 사례는 없었습니다. [출처](https://simonwillison.net/2026/Jun/26/hack-my-ai-assistant)

사용 모델은 **Claude Opus 4.6**이며, 앤트로픽이 훈련 단계에서 인젝션 저항성을 높인 결과로 분석됩니다.

다만 Simon Willison은 중요한 경계를 짚습니다. "6,000회 실패가 면역을 보장하지는 않는다"는 것입니다.

**실전 가이드라인**:

- 비가역적 작업(DB 삭제, 결제, 파일 전송)을 수행하는 에이전트에는 모델 방어에만 의존하지 말 것.
- 사용자 입력은 항상 비신뢰(untrusted) 데이터로 처리하고, 화이트리스트 기반 입력 검증을 병행할 것.
- 프롬프트 인젝션 테스트를 배포 전 QA 체크리스트에 포함할 것.

모델 방어 성능이 높아진 것은 실질적 진전이지만, 아키텍처 수준의 방어를 대체하지는 않습니다.

## 4. 프런티어 모델 출시 속도의 경제 구조: 과장된 공포와 실속 있는 판단

Dean W. Ball의 분석은 오늘 신호들을 묶는 맥락을 제공합니다. [출처](https://simonwillison.net/2026/Jun/26/dean-w-ball)

요점은 단순합니다. 프런티어 모델은 공개 직후 좁은 시간 안에 훈련 비용을 회수해야 합니다. 경쟁 모델이 마진을 잠식하기 전에 시장 점유를 확보해야 하기 때문입니다.

GPT-5.6의 빠른 프리뷰 공개, 앤트로픽 Mythos 모델의 제한 배포 구조, 안전 검토 기간 단축 압력은 모두 이 경제 논리의 표현입니다.

개발자와 강사 입장에서 두 가지 실속 있는 판단이 가능합니다.

첫째, 새 모델이 나왔을 때 "왜 이렇게 빨리 나왔나"보다 "어떤 작업에서 기존 모델을 실제로 앞서는가"를 먼저 검증하는 습관이 중요합니다. 둘째, 비용과 출시 속도 압박이 구조적으로 존재한다면, 프로덕션 시스템의 에이전트 설계에서 안전 마진을 개발자 스스로 확보해야 합니다. 오늘 나온 사고 보고서와 인젝션 실험은 그 판단을 뒷받침하는 증거입니다.
