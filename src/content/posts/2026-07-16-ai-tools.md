## 1. Claude web_fetch 탈취 취약점: 링크 체이닝 공격 구조

보안 연구자 Ayush Paul이 **Claude**의 `web_fetch` 도구에서 데이터 탈취 경로를 발견했습니다.

공격 방식은 다음 구조입니다. 공격자가 가짜 Cloudflare 인증 페이지를 만들고, 클로드가 그 페이지를 가져오면 페이지 내부 링크(`/a`, `/ay`, `/ayu` …)를 순서대로 방문하도록 유도합니다. 직접 URL에 민감 데이터를 붙이는 방식은 차단되어 있지만, 페이지 내 링크를 따라가는 동작은 허용되어 있었습니다.

이 방식으로 사용자의 이름·직장·거주지가 외부 서버로 전송되었습니다.

**실전 대응 원칙 두 가지:**

첫째, 에이전트(Agent)가 도구 출력에서 생성한 URL을 자동으로 재방문하도록 허용하지 마십시오. 방문 허용 URL은 사용자가 직접 제공하거나 전용 검색 도구 결과만으로 제한해야 합니다.

둘째, Claude 메모리 기능이 활성화된 상태에서 웹 브라우징을 함께 사용하면 위험 면적이 넓어집니다. 민감 정보를 다루는 프로젝트에서는 두 기능의 동시 사용을 검토하십시오.

강의 시 "샌드박싱이 있어도 도구 출력 체이닝은 별도로 설계해야 한다"는 점을 실제 사례로 설명하는 데 활용할 수 있습니다. [출처](https://simonwillison.net/2026/Jul/15/claude-web-fetch-exfiltration)

## 2. Grok Build CLI 오픈소스 전환: 로컬 퍼스트 전환 방법

xAI의 **Grok Build** CLI가 아파치 2.0 라이선스로 공개되었습니다.

공개 이전에는 프로젝트 디렉터리 전체를 클라우드에 업로드한다는 점이 논란이 되었습니다. 이번 공개로 코드베이스(84만 4,530줄, Rust)를 직접 감사할 수 있게 되었습니다.

로컬 퍼스트(local-first)로 전환하는 절차는 다음과 같습니다.

1. 저장소를 클론한 뒤 자체 추론 엔드포인트를 설정합니다.
2. 데이터 보존(data retention) 옵션이 기본 비활성화 상태인지 확인합니다.
3. 민감한 환경 변수나 API 키가 포함된 디렉터리는 `.gitignore`와 동일한 방식으로 제외 범위를 설정합니다.

xAI는 이전에 수집한 사용자 데이터를 삭제했다고 밝혔습니다. 코드 감사 후 사용 여부를 결정하는 것이 안전합니다. 자체 인프라에서 AI 코딩 도구를 운영해야 하는 팀에는 실용적인 선택지가 생긴 셈입니다. [출처](https://simonwillison.net/2026/Jul/15/grok-build)

## 3. Grok Mermaid 유니코드 렌더러: 터미널·문서 다이어그램 삽입

xAI가 공개한 오픈소스 메르메이드(Mermaid) 렌더러는 다이어그램 문법을 유니코드 박스 문자로 변환합니다. Simon Willison이 이를 웹어셈블리(WebAssembly)로 포팅하고 브라우저 도구로 만들었습니다.

입력은 일반 메르메이드 문법이고, 출력은 터미널에서 바로 표시되는 텍스트 박스 그림입니다.

```
graph LR
  A[입력] --> B[처리] --> C[출력]
```

위 문법이 `┌──────┐ ──→ ┌──────┐` 형태의 텍스트로 변환됩니다.

**활용 시나리오:**

- README에 이미지 없이 아키텍처 다이어그램 삽입
- CI 로그·쉘 스크립트 안에 플로우 삽입
- 이미지 렌더링이 불가능한 문서 시스템(위키, Confluence 일부 환경)

강의 자료에서 다이어그램을 코드 블록으로 넣을 때 별도 이미지 없이 표현 가능하다는 점이 실용적입니다. [출처](https://simonwillison.net/2026/Jul/16/grok-mermaid)

## 4. Cursor 제로데이: 신뢰하지 않는 저장소 실행 위험

Windows용 **Cursor**에서 저장소를 열기만 해도 임의 코드가 실행될 수 있는 제로데이 취약점이 공개되었습니다.

원인은 Cursor가 저장소 내부 디렉터리를 git 실행 파일 탐색 경로에 포함하기 때문입니다. 악성 저장소에 `git.exe`를 심어두면 사용자 확인 없이 실행됩니다.

패치가 나오기 전까지 적용할 조치:

- 출처를 확인하지 않은 저장소는 Cursor에서 열지 마십시오.
- 반드시 열어야 한다면 가상 머신 또는 격리된 환경에서 여십시오.
- 팀 내 저장소 검증 절차(리뷰·서명 확인)를 강화하십시오.

완전 공개가 선택된 이유는, 사용자의 별도 행동 없이 "열기"만으로 트리거되기 때문입니다. 취약점의 존재를 즉시 알리는 것이 유일한 방어 수단이었습니다. [출처](https://news.hada.io/topic?id=31461)

## 5. AI 서비스 신뢰도 실측 vs 광고 수치: 도구 선택 기준

AIWatch의 2026년 6월 신뢰도 보고서는 41개 AI 서비스를 대상으로 업타임·인시던트·지연시간을 측정했습니다.

Claude의 공식 업타임은 **99.55%**였지만, 6월 한 달 인시던트는 **45건**이었습니다. 두 수치의 구조적 차이를 읽어야 합니다. 높은 업타임 비율은 짧고 잦은 장애를 가릴 수 있습니다. 개발 중 짧은 장애가 반복되면 디버깅 시간과 혼동 비용이 누적됩니다.

도구 선택 시 확인할 기준:

- 업타임 퍼센트 외에 월별 인시던트 건수를 함께 확인합니다.
- 인시던트 지속 시간과 발생 시간대(업무 시간대 여부)를 봅니다.
- SLA(서비스 수준 협약)가 필요한 프로덕션 워크플로에는 복수 API를 폴백으로 구성합니다.

오늘 신호들을 묶으면 하나의 흐름이 됩니다. 클로드 탈취 취약점·Cursor 제로데이·신뢰도 실측 수치는 모두 "기능의 외관"과 "실제 보안·가용성 사이의 간극"을 드러냅니다. Grok Build 오픈소스 전환과 Mermaid 렌더러 공개는 그 간극을 직접 감사하고 제어하는 방향의 대안입니다. 도구를 고를 때 투명성 확보 가능 여부가 이제 실용적 기준이 되고 있습니다. [출처](https://news.hada.io/topic?id=31485)
