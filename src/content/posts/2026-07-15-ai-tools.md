## 1. Claude Code 스킬 묶기: 워크플로우 자동화 실전법

**Claude Code**에는 반복 프롬프트를 재사용 단위로 묶는 '스킬(Skill)' 기능이 있습니다.

Skill Creator 플러그인으로 표준 작업 단위를 한 번 정의해 두면, 이후에는 호출만으로 실행됩니다.

여러 스킬을 순서대로 묶은 '마스터 스킬'을 만들면, 복잡한 다단계 작업을 단일 명령으로 처리할 수 있습니다.

**실전 팁 세 가지.**

첫째, 스킬 템플릿에는 입출력 형식을 명시합니다. 다음 스킬이 앞 스킬의 출력을 그대로 받을 수 있어야 체인이 끊기지 않습니다.

둘째, 마스터 스킬에는 각 단계 사이 검증 조건을 넣어 두십시오. 중간 실패를 조기에 포착합니다.

셋째, 강의 시연용으로는 단일 스킬부터 보여 주고, 마스터 스킬을 '스킬의 스킬'로 설명하면 개념이 빠르게 전달됩니다.

이 구조는 반복 프롬프트 입력을 줄이고, 개발자가 실질적인 문제 해결에 집중할 시간을 확보해 줍니다. [출처](https://yozm.wishket.com/magazine/detail/3849)

## 2. Codex CLI 멀티에이전트: 서브에이전트 프롬프트 암호화

**Codex CLI**의 MultiAgentV2가 서브에이전트 프롬프트를 암호화하기 시작했습니다.

`spawn_agent`, `send_message`, `followup_task` 메시지의 위임 내용이 부모 감사 로그에서 읽히지 않습니다.

PR #26210이 2026년 6월에 병합된 이후 적용된 변경입니다.

보안 측면에서는 에이전트 간 민감한 지시 내용이 외부에 노출될 위험이 줄어듭니다.

반면, 디버깅과 규정 감사(Audit) 측면에서는 가시성이 낮아집니다.

**실전 대응책.**

멀티에이전트 워크플로우를 운영 중이라면, 암호화 전 별도 감사 시스템을 병행 구성하십시오.

규정 요건이 있는 환경에서는 전체 프롬프트 투명성이 필요한지 먼저 검토한 뒤 업그레이드를 적용하십시오.

강의에서는 "에이전트가 에이전트를 호출할 때 보안과 투명성의 트레이드오프(Trade-off)가 생긴다"는 설명 포인트로 활용할 수 있습니다. [출처](https://news.hada.io/topic?id=31449)

## 3. PostHog 사례: Claude Code로 SQL 파서를 Rust로 재작성

PostHog가 C++ SQL 파서를 **Claude Code** 다중 세션으로 Rust로 재작성해 벤치마크 기준 **70배** 속도를 달성했습니다.

새 구현은 16,000줄 Rust 코드이며, 예측적 재귀 하강 파싱과 Pratt 표현식 파서를 핵심으로 씁니다.

테스트 수천 줄과 도구 코드 5,000줄도 Claude Code 세션으로 생성했습니다.

**이 사례에서 얻을 수 있는 워크플로우 패턴.**

재작성 전 아키텍처 패턴(재귀 하강 + Pratt 파싱)을 먼저 확정하고, Claude Code에는 패턴을 명시한 뒤 세션을 시작하십시오.

각 세션은 단일 모듈 단위로 끊어서 진행하고, 세션 간 테스트를 실행해 회귀를 조기에 잡는 것이 핵심입니다.

수동 코드 리뷰 없이도 진행이 가능했다는 점은, 명확한 아키텍처 결정이 AI 지시를 대체한다는 것을 보여 줍니다.

성능이 병목인 데이터베이스 도구를 재작성할 때 이 접근법을 참고할 수 있습니다. [출처](https://news.hada.io/topic?id=31457)

## 4. 토큰맥싱 경고: 토큰 소비량은 생산성 지표가 아닙니다

'토큰맥싱'은 문제 해결보다 토큰 소비 극대화에 AI 사용을 맞추는 경향을 가리킵니다.

코드 줄 수, 스토리 포인트(Story Points)처럼 측정이 쉬운 지표가 실제 목표를 대체해 버리는 패턴의 반복입니다.

대표적인 낭비 형태는 세 가지입니다.

첫째, 프롬프트 플러딩(Prompt Flooding) — 필요 없는 맥락을 모든 요청에 과도하게 넣는 것.

둘째, 에이전트 군집 — 조율 없이 여러 에이전트를 병렬 실행하는 것.

셋째, 백그라운드 루프 — 출력 책임 없이 AI 세션을 계속 돌리는 것.

**실전 점검 질문.**

"토큰을 얼마나 썼는가?" 대신 "누구의 어떤 문제를 실제로 해결했는가?"를 물어야 합니다.

강의에서는 이 대비를 스토리 포인트 실패 사례와 함께 제시하면 설득력이 높아집니다. [출처](https://www.itworld.co.kr/article/4197014/)

## 5. 마찰은 낭비인가, 동기화 장치인가 — 오늘의 대비

Armin Ronacher는 코드 리뷰와 조율 오버헤드가 팀의 공유 이해를 유지하는 마찰이라고 주장합니다.

AI 에이전트가 이 마찰을 제거하면, 팀원 각자의 시스템 이해가 어긋나기 시작할 위험이 있다는 경고입니다.

한편, PostHog 사례는 마찰을 아키텍처 결정으로 전환했을 때 AI가 반복 작업을 흡수할 수 있음을 보여 줍니다.

오늘 신호들을 하나의 흐름으로 정리하면, AI 도구의 실질적 가치는 사용량 극대화가 아니라 **팀의 판단력과 가시성을 유지하면서 반복을 제거하는 데** 있습니다.

스킬 묶기로 반복을 줄이되, 암호화된 에이전트 로그와 토큰 소비량으로 성과를 측정하지 않도록 주의하십시오. [출처](https://simonwillison.net/2026/Jul/14/armin-ronacher)
