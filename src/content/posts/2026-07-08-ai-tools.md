## 1. sqlite-utils 4.0: Python 선언형 스키마 마이그레이션

**sqlite-utils 4.0**이 2026년 7월 7일 공개됐습니다. 가장 큰 변화는 Python 파일로 마이그레이션을 정의하는 기능입니다. [출처](https://simonwillison.net/2026/Jul/7/sqlite-utils-4)

사용법은 다음과 같습니다.

`migrations.py` 파일에 `@migrations()` 데코레이터로 변환 단계를 순서대로 선언하고, 아래 명령 한 줄로 실행합니다.

```
uvx sqlite-utils migrate data.db migrations.py
```

실행된 마이그레이션은 `_sqlite_migrations` 테이블에 기록됩니다. 중복 실행이 자동으로 차단됩니다.

내부적으로는 sqlite-utils 기존 `table.transform()` 메서드를 활용합니다. SQLite가 기본 지원하지 않는 컬럼 타입 변경·이름 변경도 이 방식으로 처리됩니다.

**실전 팁:** 마이그레이션 파일을 Git으로 버전 관리하면 개발·테스트·운영 환경의 스키마를 동일하게 유지할 수 있습니다. 기존 `sqlite-migrate` 사용자는 0.2 버전의 호환 심(shim)이 자동 처리하므로 코드 수정 없이 업그레이드 가능합니다. [출처](https://simonwillison.net/2026/Jul/7/sqlite-migrate)

4.0 출시 전 최종 릴리스 후보(RC4) 단계에서 **Claude Fable 5**의 코드 리뷰를 거쳤습니다. AI가 오픈소스 라이브러리의 주요 배포 전 감수 도구로 자리잡는 패턴을 보여주는 사례입니다. [출처](https://simonwillison.net/2026/Jul/7/sqlite-utils-2)

강의 환경에서는 학습 데이터베이스의 스키마를 수업 진행에 맞춰 단계적으로 변경하는 실습에 바로 적용할 수 있습니다.

## 2. Claude Code 루프 패턴: 에이전트 자율 반복 분류법

**Claude Code** 팀이 에이전트 루프(agentic loop) 패턴을 네 가지 기준으로 분류한 자료를 공개했습니다. [출처](https://news.hada.io/topic?id=31225)

분류 기준은 다음 네 가지입니다.

- **트리거 방식**: 사람이 시작하는가, 이벤트가 시작하는가
- **종료 조건**: 성공 조건 도달, 오류 발생, 반복 횟수 초과
- **사용 프리미티브**: 도구 호출, 파일 읽기/쓰기, 외부 API
- **적합한 태스크 유형**: 반복 디버깅, 테스트 생성, 점진적 리팩터링

이 분류를 알면 루프 설계 시 종료 조건을 먼저 정의하는 습관이 생깁니다. 종료 조건이 모호한 루프는 토큰을 무한 소모하거나 잘못된 상태로 수렴할 위험이 있습니다.

**실전 팁:** 테스트 자동 생성처럼 반복 횟수가 예측 가능한 태스크에는 횟수 기반 종료를 쓰고, 디버깅처럼 성공 여부가 불확실한 태스크에는 조건 기반 종료를 명시적으로 프롬프트에 포함하는 것이 안정적입니다.

강의에서는 이 네 가지 기준을 체크리스트로 제공하면 수강생이 에이전트 설계를 구조적으로 접근하는 데 도움이 됩니다.

## 3. Gemini API 관리형 에이전트: 백그라운드 태스크·원격 MCP 확장

Google이 **Gemini API** 관리형 에이전트(Managed Agent)에 백그라운드 태스크 실행과 원격 MCP(Model Context Protocol) 연동을 추가한다고 발표했습니다. [출처](https://blog.google/innovation-and-ai/technology/developers-tools/expanding-managed-agents-gemini-api)

백그라운드 태스크는 사용자 응답을 기다리지 않고 장시간 작업을 비동기로 실행할 수 있도록 합니다. 원격 MCP 연동은 에이전트가 외부 도구 서버와 표준 프로토콜로 통신하는 구조입니다.

**실전 팁:** 원격 MCP를 쓰면 에이전트 코드와 도구 구현을 분리할 수 있습니다. 도구 서버를 독립적으로 배포하고 버전 관리하는 구조가 가능해집니다. 기존 Claude MCP 서버를 Gemini 에이전트에 연결하는 실험도 가능해질 전망입니다.

공식 문서가 아직 제한적이므로, 현재는 기능 조합의 아키텍처 설계 단계에서 참고하는 수준으로 접근하는 것이 적절합니다.

## 4. github-code 웹 컴포넌트: GPT-5.5로 만든 코드 임베드 도구

Simon Willison이 **GPT-5.5**를 활용해 `<github-code>` 웹 컴포넌트(Web Component)를 제작했습니다. [출처](https://simonwillison.net/2026/Jul/7/github-code-component)

사용법은 HTML에 태그 하나를 삽입하는 것입니다.

```html
<github-code href="https://github.com/user/repo/blob/hash/file.py#L9-L18">
</github-code>
```

컴포넌트가 GitHub URL을 파싱해 `raw.githubusercontent.com` 엔드포인트를 호출하고, 지정한 줄 범위만 줄 번호와 함께 표시합니다.

**실전 팁:** 문서나 블로그에서 코드 예제를 수동 복사하면 소스와 내용이 어긋날 수 있습니다. 이 컴포넌트를 쓰면 소스 파일이 업데이트될 때 임베드 내용도 자동으로 따라갑니다. 외부 의존성이 없고 순수 HTML에서 동작합니다.

이 사례 자체가 GPT-5.5로 실용 도구를 빠르게 프로토타입하는 방식을 보여주는 예시로, 강의 데모 소재로도 쓸 수 있습니다.

## 5. 가트너 예측 vs 현실: 2029년 초소형 팀 60% 전환 논쟁

가트너(Gartner)는 2029년까지 기업의 60%가 2~5인 규모의 초소형 팀을 도입할 것으로 전망했습니다. [출처](https://www.itworld.co.kr/article/4194153/ai%ea%b0%80-%ec%86%8c%ed%94%84%ed%8a%b8%ec%9b%a8%ec%96%b4-%ec%97%94%ec%a7%80%eb%8b%88%ec%96%b4%eb%a7%81-%ec%9e%ac%ec%a0%95%ec%9d%98%ec%b4%88%ec%86%8c%ed%98%95-%ed%8c%80-%eb%8f%84%ec%9e%85.html)

이 전망의 실속 있는 부분과 과장 가능성을 나눠 보면 다음과 같습니다.

실속 있는 부분은 셋입니다. 첫째, AI가 반복 기술 작업을 흡수하면서 소수 인원이 더 넓은 범위를 담당하는 구조 자체는 이미 진행 중입니다. 둘째, 플랫폼 엔지니어링과 셀프서비스 도구 없이는 초소형 팀이 운영되기 어렵습니다. 셋째, 주니어 채용 축소는 2028년 이후 시니어 공급 부족으로 이어질 수 있다는 경고는 근거 있는 지적입니다.

과장 가능성은 하나입니다. 60%라는 수치는 '도입'의 정의에 따라 크게 달라집니다. 파일럿 팀 하나를 운영하는 것도 집계될 수 있습니다.

오늘 신호를 종합하면, 도구 층위에서는 sqlite-utils의 마이그레이션 기능처럼 실무 공백을 채우는 작고 확실한 업데이트가 이어지고 있습니다. 조직 층위에서는 AI 효율이 팀 구조 변화로 전환되는 속도를 두고 전망과 현실이 아직 간격을 유지하고 있습니다.
