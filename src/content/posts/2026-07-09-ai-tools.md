## 1. Bun Zig→Rust 재작성: AI 에이전트 + 테스트 하네스

JavaScript 런타임 **Bun**이 기존 Zig 코드베이스를 Rust로 전면 재작성했습니다. 이 작업의 핵심은 AI 에이전트가 기존 TypeScript 테스트 스위트를 실시간 적합성(Conformance) 기준으로 삼아 코드를 점진적으로 포팅했다는 방식입니다. [출처](https://simonwillison.net/2026/Jul/8/rewriting-bun-in-rust)

Zig는 가비지 컬렉션과 수동 메모리 관리를 혼용할 때 use-after-free, double-free 오류가 잦았습니다. Rust 컴파일러는 이를 빌드 단계에서 차단하므로, 버그 백로그의 상당 부분이 구조적으로 해소되었습니다.

**에이전트 하네스 구성 방식**은 다음과 같습니다.

- 기존 테스트 스위트를 그대로 정답 기준으로 유지합니다.
- 에이전트가 파일 단위로 Rust 변환을 시도하고, 테스트를 즉시 실행합니다.
- 실패 시 에이전트가 오류 메시지를 읽고 재시도합니다.
- 사람은 전체 통과 여부만 최종 확인합니다.

실전 팁: 이 패턴은 "테스트 커버리지가 충분히 높은 프로젝트"라는 전제 조건이 결정적입니다. 커버리지가 낮으면 에이전트가 통과 여부를 판단할 기준 자체가 없어, 오히려 잘못된 확신을 줄 수 있습니다.

강의나 실무에서 "절대 못 건드리는 레거시"로 여겼던 프로젝트도, 테스트를 먼저 보강한 뒤 에이전트 하네스를 적용하면 대규모 마이그레이션의 문턱이 낮아집니다.

## 2. GPT-Live: 풀듀플렉스 음성 모델 구조와 실전 활용

OpenAI가 **GPT-Live-1**과 GPT-Live mini를 공개했습니다. 기존 GPT-4o 기반 음성 모드는 턴-테이킹(turn-taking) 구조였지만, 이번 모델은 오디오를 양방향 동시 처리하는 풀듀플렉스 구조로 전환했습니다. [출처](https://simonwillison.net/2026/Jul/8/introducing-gptlive)

구조적 차이는 다음과 같습니다.

- **이전 모델**: 사용자가 말을 마친 뒤 AI가 응답합니다.
- **GPT-Live**: 사용자가 말하는 중에도 AI가 컨텍스트를 처리합니다. 자연스러운 끼어들기(interjection)와 생각 중 멈춤(pause)이 가능합니다.

복잡한 작업은 GPT-5.5가 백그라운드에서 처리하고, 대화 흐름은 별도로 유지됩니다. 연산 중에도 사용자와 대화가 끊기지 않는 구조입니다.

실전 팁: 손을 쓸 수 없는 상황(걸으면서 아이디어 정리, 코드 리뷰 브리핑 청취 등)에서 활용 폭이 늘어납니다. 음성 기반 튜터링 앱을 개발할 때는 풀듀플렉스 API 엔드포인트와 턴-테이킹 엔드포인트를 구분해 설계해야 UX가 자연스럽습니다.

## 3. GitHub 에이전틱 워크플로: 교차 레포 문서화 + 보안 취약점

GitHub 에이전틱 워크플로는 병합된 PR 변경 사항을 감지해, 별도 문서 레포지토리에 PR을 자동 생성하고 담당 전문가(SME) 리뷰를 요청하는 문서화 파이프라인을 지원합니다. Aspire 팀이 릴리스 싱크 용도로 실제 운영 중입니다. [출처](https://github.blog/ai-and-ml/github-copilot/automating-cross-repo-documentation-with-github-agentic-workflows)

설정 순서는 간단합니다.

1. 메인 레포에서 PR 병합 이벤트를 트리거로 등록합니다.
2. 변경 내용을 추출해 문서 레포에 초안 PR을 생성합니다.
3. 해당 PR에 SME 레이블과 리뷰어를 자동 지정합니다.

그런데 같은 날, **GitLost**라는 간접 프롬프트 인젝션(Indirect Prompt Injection) 취약점이 보고됐습니다. 공개 이슈 댓글에 악성 마크다운을 삽입하면, 에이전트가 같은 조직의 비공개 레포 데이터를 유출할 수 있습니다. [출처](https://news.hada.io/topic?id=31255)

즉각 적용 가능한 대응책은 세 가지입니다.

- 워크플로 권한을 최소 필요 범위로 제한합니다.
- 이슈 템플릿에서 외부 입력을 사전 검증합니다.
- 공개 레포 컨텍스트와 비공개 레포 컨텍스트를 워크플로 내에서 격리합니다.

GitHub의 패치가 배포되기 전까지는 공개 이슈를 처리하는 워크플로에 비공개 레포 접근 권한을 부여하지 않는 것이 가장 안전한 임시 조치입니다.

## 4. Weave Router + Kokoro: 멀티 LLM 라우팅과 로컬 TTS 실용 조합

**Weave Router**는 Anthropic, OpenAI, Gemini를 단일 엔드포인트로 통합하는 드롭인 프록시입니다. 요청마다 최적 모델을 자동 선택하며, 라우팅 지연은 50ms 이내, 비용은 40~70% 절감 효과가 보고됩니다. 기존 API 호출 코드에서 엔드포인트 URL만 교체하면 됩니다. [출처](https://news.hada.io/topic?id=31251)

같은 날 **Kokoro**도 주목할 만합니다. 파라미터 수 8,200만 개(82M)의 TTS 모델로, GPU 없이 로컬 CPU에서 고품질 음성을 생성합니다. 영어, 중국어, 힌디어를 지원하며 약 50가지 음성을 제공합니다. [출처](https://news.hada.io/topic?id=31239)

두 도구를 조합하면 실용적인 로컬 AI 보이스 스택이 구성됩니다.

- GPU: LLM 추론에 전담 할당합니다.
- CPU: Kokoro가 TTS를 처리합니다.
- API 비용: Weave Router가 모델 선택으로 절감합니다.

오프라인 음성 앱이나 GPU 자원이 제한된 환경에서 LLM+TTS를 동시에 운용해야 할 때 구조적으로 유리한 조합입니다.

## 5. AI 코드 리뷰 관행 논쟁: PR 메시지를 AI에 맡기면 생기는 구조적 문제

Cloudflare Workers 런타임 개발자 Kenton Varda가 AI 생성 PR 메시지와 커밋 설명에 대한 사용 금지 방침을 선언했습니다. [출처](https://simonwillison.net/2026/Jul/8/kenton-varda)

핵심 문제는 구조적입니다. AI는 diff에서 읽을 수 있는 변경 사항(무엇이 바뀌었는가)은 잘 요약합니다. 그러나 "왜 이 방식을 선택했는가"라는 의도와 맥락은 diff에 없기 때문에, AI 요약이 오히려 고수준 이해를 방해합니다.

이 구조는 Bun 재작성 사례와 정반대의 교훈을 줍니다. 테스트 통과처럼 기계적으로 검증 가능한 작업은 에이전트가 유리합니다. 의도와 판단이 필요한 서술적 작업은 사람이 직접 써야 효과적입니다.

오늘 신호들을 관통하는 흐름은 하나입니다. AI 에이전트는 검증 기준이 명확하고 반복 가능한 구간에서 점점 더 큰 역할을 맡고 있습니다. 반면 그 경계 밖, 즉 의도 서술·보안 경계·권한 설계는 여전히 사람의 판단이 구조를 결정합니다.
