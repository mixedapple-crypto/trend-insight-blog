## 1. Vercel eve 프레임워크: 스킬·샌드박스·에이전트 가독 웹

**Vercel**의 Andrew Qu가 eve 에이전트 프레임워크의 설계 원칙을 공개했습니다. [출처](https://www.latent.space/p/vercel-agents-new-software)

핵심 구성 요소는 세 가지입니다.

첫째, **스킬(Skill)**은 타입이 지정된 함수로 정의합니다. 에이전트가 이를 탐색하고 체이닝(chaining)하므로 순차 워크플로를 하드코딩할 필요가 없습니다.

둘째, 샌드박스(Sandbox)는 실행 환경을 격리합니다. 에이전트가 예측 불가능한 추론 루프에 빠지지 않고 제한된 범위 안에서 동작하게 합니다.

셋째, 에이전트 가독 웹사이트는 시맨틱 HTML이나 구조화된 API 형태로 데이터를 제공합니다. 자유형 브라우징보다 도구 사용 신뢰성이 높습니다.

실전 적용 포인트는 다음과 같습니다. 기존 "챗봇형 에이전트" 설계를 벗어나, 스킬을 독립적이고 조합 가능한 단위로 먼저 정의하십시오. 에이전트가 스킬 목록을 보고 스스로 순서를 결정하게 하면 프롬프트 복잡도가 줄어듭니다. 강의 맥락에서는 "에이전트 = 스킬 조합 소프트웨어"라는 프레임이 학습자의 설계 사고를 구조화하는 데 유효합니다.

## 2. llm-coding-agent 0.1a0: 승인 게이트 내장 코딩 에이전트 라이브러리

Simon Willison이 `llm-coding-agent 0.1a0`를 공개했습니다. [출처](https://simonwillison.net/2026/Jul/2/llm-coding-agent)

파일 편집·명령 실행·테스트 주도 개발 워크플로를 지원하는 Python 라이브러리입니다.

**설치 및 실행 방법**은 아래와 같습니다.

```bash
# 설치
uvx --prerelease=allow --with llm-coding-agent llm code

# 자동 승인 모드
llm code --yolo

# 특정 명령만 허용
llm code --allow "pytest*"
```

Python API로도 사용할 수 있습니다.

```python
CodingAgent(
  model="gpt-5.5",
  root="/path",
  approve=True
).run("테스트 실패 수정해줘")
```

핵심 도구는 두 가지입니다. `CodingTools_edit_file()`은 문자열 대체와 diff 검증을 함께 수행합니다. `CodingTools_execute_command()`는 셸 명령을 실행하되 타임아웃을 120초로 제한합니다.

실전 팁은 다음과 같습니다. `--yolo` 모드는 CI 파이프라인에서, `--allow` 모드는 로컬 개발 중 인간 검토가 필요한 단계에 각각 구분해 사용하십시오. 리팩터링·테스트 수정·기능 추가처럼 다단계 코딩 작업을 자동화할 때 승인 게이트가 오류 누적을 막는 핵심 안전장치입니다.

## 3. DSPy로 프롬프트 병목 찾기: SQL 에이전트 개선 사례

Simon Willison이 **Claude Code**를 활용해 DSPy(프롬프트 최적화 프레임워크)로 Datasette 에이전트의 SQL 시스템 프롬프트를 평가했습니다. [출처](https://simonwillison.net/2026/Jul/2/dspy-datasette-agent-prompts)

발견한 구체적 문제는 이것입니다. 프롬프트에 "describe_table을 호출하지 말라"는 지시가 있었는데, 이 때문에 에이전트가 컬럼명을 추측하다가 오류·재시도 루프에 빠졌습니다.

해결책은 간단합니다. 스키마 목록에 컬럼명을 포함시키면 루프가 사라집니다.

이 사례의 가치는 방법론에 있습니다. DSPy의 평가 기능을 쓰면 수동 시행착오 없이 어떤 컨텍스트 정보가 환각을 줄이는지 자동으로 찾아낼 수 있습니다.

데이터베이스나 API를 조회하는 에이전트를 만드는 실무자라면, 프롬프트 개선을 직관이 아닌 평가 지표로 접근하는 이 워크플로가 즉시 참고할 수 있는 패턴입니다.

## 4. "이해하며 참여하라": 에이전트 협업의 인지 부채 문제

Geoffrey Litt가 제시한 "understand to participate(이해하며 참여하라)" 원칙이 실무 가이드로 주목받고 있습니다. [출처](https://simonwillison.net/2026/Jul/2/understand-to-participate)

에이전트가 대규모 다중 파일 리팩터링을 처리할 때, 개발자가 출력을 수동적으로 수용하면 **인지 부채(cognitive debt)**가 쌓입니다. 코드의 실제 구조와 개발자의 정신 모델 사이 간극이 벌어지는 것입니다.

핵심 사고방식의 차이는 다음과 같습니다.

- 수동적 관찰자: 에이전트 출력을 승인만 함
- 능동적 협력자: 핵심 개념에 대한 이해를 유지하며 방향을 조정함

실전 팁입니다. 에이전트 작업 중간에 체크포인트를 설정하십시오. "이 변경이 왜 이 구조를 선택했는가"를 스스로 설명할 수 없다면 다음 단계로 넘기기 전에 이해를 먼저 회복하십시오.

강사라면 이 원칙을 "에이전트는 대화 상대"라는 수업 메시지로 직접 쓸 수 있습니다. 에이전트를 잘 쓰려면 도메인 지식이 오히려 더 중요해집니다.

## 5. Fable 5 재공개 축소·벤치마크·Shopify 설계 원칙: 실속 정리

**Fable 5**와 Mythos 5 모델이 미국 수출 규제로 인한 3주 중단 후 재공개되었습니다. [출처](https://www.itworld.co.kr/article/4192513/)

단, 구독자 조건이 대폭 줄었습니다. 이용 기간은 2주에서 1주(7월 7일까지)로 단축됐고, 사용량은 평소의 50%로 제한됩니다. 7월 7일 이후 계속 쓰려면 별도 크레딧을 구매해야 하며, 일반 코딩·디버깅 작업은 Opus 4.8로 자동 전환됩니다.

**CursorBench 3.1** 평가 결과에서는 Fable 5 Max가 72.9%로 선두를 기록했습니다. [출처](https://news.hada.io/topic?id=31074) Opus 4.7 Max는 64.8%, GPT-5.5는 그 아래입니다. Cursor 사용자라면 복잡한 리팩터링에는 Fable Max 티어를, 린팅이나 단순 완성 작업에는 하위 티어를 배분하는 예산 설계가 가능합니다.

Shopify의 River 사례는 에이전트 도입보다 작업장 설계가 먼저라는 점을 보여줍니다. [출처](https://www.itworld.co.kr/article/4192545/) River가 슬랙 공개 채널에서만 동작하도록 강제한 것, 모노레포와 Nix로 개발 환경을 표준화한 것이 에이전트 성능을 끌어올린 실제 원인입니다.

오늘 신호들이 가리키는 방향은 하나로 수렴합니다. 더 강한 모델보다 더 잘 설계된 스킬·환경·프로세스가 에이전트의 실제 성능을 결정한다는 것입니다. Fable 5 접근 조건 축소는 이 판단을 모델 의존도를 낮추는 쪽으로 재점검할 계기이기도 합니다.
