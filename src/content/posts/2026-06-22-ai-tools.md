## 1. sqlite-utils 4.0rc1: 마이그레이션 시스템 + 중첩 트랜잭션

**sqlite-utils 4.0rc1**이 2026년 6월 21일 공개됐습니다.
핵심은 두 가지, 마이그레이션(migration) 시스템과 중첩 트랜잭션(nested transaction)입니다.

마이그레이션은 별도 패키지였던 `sqlite-migrate`를 본체에 흡수한 결과입니다.
`migrations.py` 파일에 `@migrations()` 데코레이터로 함수를 정의하면,
테이블 생성·컬럼 추가 같은 스키마 변경을 코드로 선언할 수 있습니다.

```python
# migrations.py 예시
from sqlite_utils import migrations

@migrations()
def add_species_column(db):
    db["creatures"].add_column("species", str)
```

적용은 Python에서 `migrations.apply(db)`,
CLI에서는 `sqlite-utils migrate creatures.db migrations.py`로 실행합니다.

롤백(rollback) 기능은 의도적으로 제외했습니다.
실수는 새 순방향 마이그레이션으로 교정하는 설계입니다.
이 방식은 이미 **LLM** 프로젝트 등 실 서비스에서 수년간 검증됐습니다.

중첩 트랜잭션은 세이브포인트(savepoint) 기반으로 동작합니다.
기존 트랜잭션 안에서 부분 실패 시 중간 작업을 보존할 수 있어,
멀티스텝 DB 작업의 안정성이 높아집니다.

강의나 개발 모두에서 즉시 활용 가능합니다.
Claude Code·LLM CLI처럼 SQLite를 내부 저장소로 쓰는 AI 도구와 함께 쓰면,
스키마 진화를 코드로 추적하고 리뷰할 수 있는 구조를 갖출 수 있습니다. [출처](https://simonwillison.net/2026/Jun/21/sqlite-utils-40rc1)

## 2. Claude Code 생태계: Recall 로컬 메모리 + myDoo IDE

Claude Code 주변에서 두 도구가 동시에 등장했습니다.

**Recall**은 Claude Code 세션 간 프로젝트 컨텍스트(context)를 로컬 DB에 저장하는 도구입니다.
설치는 `npm install -g recall`, 이후 `recall index`로 프로젝트를 스캔하고,
세션 시작 전 `recall query <질문>`으로 관련 컨텍스트를 미리 꺼냅니다.

외부 서버로 데이터가 나가지 않고, 토큰 사용량도 줄어듭니다.
매 세션마다 "이 프로젝트는 이런 구조입니다"를 반복 설명하는 비용을 줄이는 구조입니다. [출처](https://github.com/raiyanyahya/recall)

**myDoo**는 VS Code 기반 IDE로, Claude Code를 기본 모델로 탑재했습니다.
코딩 프로젝트와 문서 프로젝트를 분리해 컨텍스트 전환을 지원합니다.
별도 채팅 인터페이스 없이 에디터 안에서 Claude 기반 지원을 바로 씁니다. [출처](https://news.hada.io/topic?id=30709)

두 도구 모두 "Claude Code를 반복적으로 쓰는 사람"을 겨냥합니다.
Recall은 세션 지속성, myDoo는 편집 환경 통합이라는 서로 다른 문제를 각각 풉니다.
둘을 병행하면 컨텍스트 관리와 편집기 워크플로를 동시에 정비할 수 있습니다.

## 3. opencodex: Codex에서 Claude·GLM 등 타 LLM 쓰는 법

**opencodex**는 Codex 환경에서 OpenAI 모델 이외의 LLM을 쓸 수 있게 하는 로컬 프록시(proxy)입니다.

Codex는 기본적으로 OpenAI 모델만 지원합니다.
opencodex는 Codex와 LLM 제공자 사이의 요청을 가로채 프로토콜을 실시간 변환합니다.
스트리밍, 툴 콜(tool call), 추론 토큰, 이미지 처리를 양방향으로 지원합니다.

실전 시나리오는 단순합니다.
Codex 워크플로를 그대로 유지하면서 Claude나 GLM-5.2 같은 모델로 교체하고 싶을 때,
프록시 하나로 즉시 전환할 수 있습니다.
OpenAI의 공식 지원을 기다릴 필요가 없습니다. [출처](https://news.hada.io/topic?id=30685)

팀이 Codex 환경에 고착돼 있지만 모델 선택 유연성이 필요한 경우,
프록시 레이어를 도입하는 이 패턴은 다른 도구에도 응용 가능합니다.

## 4. Samsung·Noam Shazeer: 엔터프라이즈 AI 코딩 도입 구조 변화

**삼성전자**가 ChatGPT 엔터프라이즈(Enterprise)와 Codex를 전사 도입했습니다.
GPT-4 및 코드 생성 모델을 내부 시스템과 연동하되,
보안·데이터 격리·컴플라이언스(compliance) 통제를 전제로 설계했습니다. [출처](https://openai.com/index/samsung-electronics-chatgpt-codex-deployment)

이 사례에서 구조적으로 주목할 점은 민감 데이터 격리 패턴입니다.
AI 도구 도입 규모가 커질수록, 도구 선택보다 데이터 흐름 설계가 먼저입니다.

같은 날 OpenAI는 트랜스포머(Transformer) 논문 공동 저자 **노암 샤지어(Noam Shazeer)**의 합류를 발표했습니다.
"10년을 기다렸다"는 샘 올트먼(Sam Altman)의 발언은 신호가 큽니다.
다만 OpenAI의 2025년 영업 손실이 208억 9,200만 달러에 달하는 맥락을 함께 읽어야 합니다. [출처](https://www.itworld.co.kr/article/4187504)

## 5. AI 인지 부채 vs. 실속: 코드 품질 데이터로 읽는 오늘의 흐름

GitClear가 2억 1,100만 건의 코드 변경을 분석한 결과,
2024년 중복 코드 블록이 8배 증가했고 리팩터링 비율은 25%에서 10% 미만으로 떨어졌습니다.
구글 DORA 보고서는 AI 도입률 25% 증가와 배포 안정성 7.2% 하락을 함께 보고했습니다.
원인은 AI 코드 자체의 결함보다, AI가 유도하는 대규모 일괄 변경(batch changeset)의 위험입니다. [출처](https://www.itworld.co.kr/article/4187513)

대응 방향은 이미 데이터로 나와 있습니다.
AI 도구에 코드베이스 아키텍처 결정, 과거 PR(풀 리퀘스트), 보안 정책, 모듈 패턴을 컨텍스트로 제공할수록 품질 격차가 좁혀집니다.
설문에서 개발자의 65%가 AI가 이 컨텍스트를 놓쳤다고 답했습니다.

오늘 등장한 도구들의 흐름을 한 줄로 묶으면 이렇습니다.
sqlite-utils의 마이그레이션, Recall의 로컬 메모리, opencodex의 프록시 모두 "AI 도구에 더 많은 컨텍스트를 공급하는 구조"를 향하고 있습니다.
과장 없이, 그 방향이 오늘의 실속입니다.
