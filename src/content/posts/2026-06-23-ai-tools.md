## 1. 프롬프트 인젝션 연구: 서식이 의미를 이긴다

Ye, Cui, Hadfield-Menell의 연구를 Simon Willison이 6월 22일 정리했습니다. [출처](https://simonwillison.net/2026/Jun/22/prompt-injection-as-role-confusion)

핵심 발견은 단순합니다. LLM은 시스템 프롬프트와 사용자 입력을 '의미'로 구분하지 않습니다. **서식 스타일**—내부 사고 블록(thinking block)처럼 보이는 형식—을 역할 경계의 신호로 사용합니다.

실험 결과가 명확합니다. 주입된 악성 텍스트가 내부 사고 블록 형식을 모방하면 공격 성공률이 **61%**였습니다. 동일한 내용을 중립적 서식으로 바꾸면(인간 눈에는 같아 보임) 성공률이 **10%**로 떨어졌습니다.

개발자가 바로 적용할 수 있는 실전 팁은 세 가지입니다.

첫째, 시스템 프롬프트에서 특수 서식 블록을 과도하게 사용하지 않습니다. 모델이 그 패턴을 '권한 있는 명령'으로 학습했을 가능성이 있습니다.

둘째, 외부 입력을 처리하는 파이프라인에서 입력 텍스트의 서식을 정규화(탈스타일화)하는 단계를 추가합니다. 내용 필터링만으로는 부족합니다.

셋째, RAG(검색 증강 생성)나 툴 호출 결과처럼 외부 텍스트가 모델에 들어오는 경로를 별도 태그로 명시적 래핑하는 구조가 방어력을 높입니다.

강의에서는 "프롬프트 보안 = 콘텐츠 필터링"이라는 단순화를 수정할 필요가 있습니다. 서식 자체가 공격 벡터라는 점을 실습 예제로 보여주면 이해도가 높아집니다.

## 2. Claude Code 실전: 브라우저에서 이미지 인페인팅 모델 이식

Simon Willison은 **Moebius 0.2B** 이미지 인페인팅(inpainting) 모델을 PyTorch·CUDA 없이 브라우저에서 동작하도록 이식했습니다. [출처](https://simonwillison.net/2026/Jun/22/porting-moebius)

작동 방식은 간단합니다. 이미지를 열고, 지울 영역을 드래그로 선택하고, "Run inpaint"를 클릭합니다. **WebGPU**를 통해 클라이언트 단에서만 실행됩니다.

주목할 점은 워크플로입니다.

Willison은 Claude.ai로 먼저 모델 이식 가능성과 가중치 파일 위치를 조사했습니다. 이후 **Claude Code**를 터미널에 띄워, 본인이 다른 Datasette 작업을 하는 동안 병렬로 이식 작업을 맡겼습니다.

이 패턴이 실전에서 유용한 이유가 있습니다. Claude Code는 다른 에이전트(Agent)나 빌드가 끝나기를 기다리는 유휴 시간 동안 별도 리팩터링이나 이식 작업을 수행할 수 있습니다. "대기 중에 맡기기" 워크플로입니다.

강의 시연 소재로도 적합합니다. 브라우저에서 바로 작동하는 [데모](https://simonw.github.io/moebius-web/)를 보여주면 "AI 코딩 도구가 실제로 무엇을 만드는가"를 직관적으로 전달할 수 있습니다.

## 3. OpenAI Daybreak: 조직 단위 보안 자동화 도구

OpenAI가 6월 22일 보안 도구 묶음 **Daybreak**를 공개했습니다. [출처](https://openai.com/index/daybreak-securing-the-world)

두 가지 핵심 도구가 있습니다.

**Codex Security**는 저장소를 스캔해 취약점 후보를 찾고 수정 패치를 생성합니다. **GPT-5.5-Cyber**는 식별된 취약점이 실제로 악용 가능한지 검증하고 조치 방향을 제시합니다.

CI/CD 파이프라인에 API로 연결해 지속적 스캔을 자동화할 수 있습니다. 수동 코드 리뷰 병목을 줄이고, 보안 엔지니어가 복잡한 신규 위협에 집중하도록 루틴 트리아지(triage) 작업을 위임하는 구조입니다.

같은 날 발표한 **Patch the Planet**은 오픈소스 메인테이너를 위한 프로그램입니다. [출처](https://openai.com/index/patch-the-planet) 프로젝트를 등록하면 AI 스캔 후 전문가 검토를 거쳐 패치를 제안합니다. 리소스가 부족한 오픈소스 팀에 실질적으로 유용한 구조입니다.

## 4. Agent-Blackbox: Claude Code 토큰 낭비 진단 도구

**Agent-Blackbox**는 Claude Code와 OpenCode 실행을 로컬에서 모니터링하는 도구입니다. [출처](https://news.hada.io/topic?id=30719)

세션 맵과 컨텍스트(context) 효율 점수로 토큰 사용 패턴을 시각화합니다.

이 도구가 필요한 이유가 수치로 드러납니다. AI 모델이 스스로 추정하는 토큰 소비량과 실제 비용의 상관계수는 **0.39**에 불과합니다. 모델 자체 예측을 믿기 어렵다는 의미입니다.

실전 활용법은 간단합니다. 멀티 스텝 코딩 작업 후 로그를 분석해 어느 단계에서 토큰이 몰리는지 파악합니다. 이를 바탕으로 컨텍스트 관리 방식과 프롬프트 구조를 조정하면 불필요한 비용을 줄일 수 있습니다.

Claude Code를 수업에서 다루는 강사라면 "토큰 비용 예측"을 실습 주제로 삼을 수 있습니다. 자동 추정과 실제 측정값의 차이를 보여주는 것 자체가 좋은 교육 소재입니다.

## 5. 멀티 모델 전략 vs 오픈 모델 전환: 두 흐름의 교차점

오늘 두 가지 상반된 조언이 동시에 들어왔습니다.

하나는 "멀티 모델 전략으로 벤더 종속을 막아라"입니다. [출처](https://www.itworld.co.kr/article/4188199/%eb%8b%a4%ec%a4%91-%eb%aa%a8%eb%8d%b8%eb%a1%9c-ai-%ec%97%85%ec%b2%b4-%ec%a2%85%ec%86%8d-%eb%a7%89%ec%95%84%eb%9d%bc-%ec%a0%84%eb%ac%b8%ea%b0%80%ec%9d%98-%ed%95%9c%eb%aa%a9%ec%86%8c%eb%a6%ac.html) Gartner는 긴 문서 처리는 Claude, 빠른 요약은 Copilot처럼 용도별로 나눠 쓰되 단일 게이트웨이로 통합하라고 권고합니다.

다른 하나는 "오픈 모델로 전환해도 단점이 크지 않다"입니다. [출처](https://news.hada.io/topic?id=30727) Llama, Mistral 등 오픈 모델의 품질 격차가 좁아진 지금, 비용과 통제권 이점이 종속 위험보다 크다는 주장입니다.

두 흐름이 공유하는 핵심은 같습니다. 단일 벤더·단일 모델에 의존하는 구조를 피하라는 것입니다.

실무적 판단 기준은 명확합니다. 복잡한 코드 생성이나 고품질 출력이 필요한 작업에는 Claude Opus처럼 검증된 상용 모델을 씁니다. GLM-5.2처럼 MIT 라이선스 오픈 모델은 비용 민감 작업이나 로컬 파인튜닝(fine-tuning)이 필요한 경우에 적합합니다. [출처](https://news.hada.io/topic?id=30728)

오늘 신호들을 하나로 묶으면 이렇습니다. 보안은 서식 수준까지 내려가야 하고, 비용은 실측해야 하며, 모델 선택은 용도와 통제권 기준으로 분리해야 합니다.
