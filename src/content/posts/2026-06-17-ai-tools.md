## 1. Claude Code + 깃 워크트리: 병렬 세션 워크플로

**Claude Code** 수석 디자이너가 Excalidraw에 자동완성 기능을 추가하는 과정을 영상으로 공개했습니다. [영상](https://news.hada.io/topic?id=30562)

핵심은 브랜치마다 별도 워크트리를 만들고, 각 디렉터리에서 Claude Code 세션을 독립 실행하는 구조입니다.

워크플로는 세 단계입니다.

첫째, 기능 단위로 워크트리를 분리합니다.

```
git worktree add ../project-autocomplete feature/autocomplete
git worktree add ../project-ui feature/ui-refactor
```

둘째, 각 디렉터리에서 Claude Code를 별도 터미널로 실행합니다. 세션이 서로 다른 브랜치를 바라보므로 충돌 없이 병렬 진행됩니다.

셋째, 한 세션이 빌드·테스트를 돌리는 동안 다른 세션에서 다음 컴포넌트를 작업합니다.

실전 팁은 두 가지입니다. 워크트리 디렉터리명에 브랜치 목적을 그대로 쓰면 세션 전환 시 혼선이 줄어듭니다. 또한 세션마다 컨텍스트 창을 독립적으로 유지하므로, 공통 유틸 변경은 먼저 main에 머지한 뒤 각 워크트리에서 `git rebase`로 받아오는 순서를 지키는 것이 좋습니다.

강의 시연에서는 "병렬로 두 가지를 만들어 보여준다"는 구조가 학습 밀도를 높입니다. 개발 실무에서는 피처 브랜치가 많은 스프린트 중반에 특히 유효합니다.

## 2. 깃 워크트리 기초: 브랜치 동시 작업 설정법

GitHub 블로그가 깃 워크트리(Git Worktree) 사용법을 정리했습니다. [출처](https://github.blog/ai-and-ml/github-copilot/what-are-git-worktrees-and-why-should-i-use-them)

기본 명령은 단순합니다.

```
git worktree add <경로> <브랜치명>
git worktree list       # 현재 워크트리 목록 확인
git worktree remove <경로>  # 작업 완료 후 정리
```

워크트리는 스태시(stash)나 리베이스 없이 브랜치를 오가는 방법입니다. 핫픽스를 긴급 처리하면서 피처 브랜치 코드를 건드리지 않을 수 있습니다.

실전 팁: Claude Code나 **GitHub Copilot** 같은 AI 코딩 도구는 현재 열린 디렉터리를 컨텍스트로 삼습니다. 워크트리를 분리하면 AI가 다른 브랜치 코드를 혼입할 가능성이 줄어듭니다. AI 코딩 도구 활용이 많은 팀일수록 워크트리 병렬화의 효과가 큽니다.

## 3. OpenAI 배포 시뮬레이션: 출시 전 모델 동작 검증

**OpenAI**가 배포 시뮬레이션(Deployment Simulation) 기능을 공개했습니다. [출처](https://openai.com/index/deployment-simulation)

실제 사용자 대화 데이터셋을 업로드하면, 새 모델이 해당 시나리오에서 어떻게 응답하는지 사전 검증하는 도구입니다.

사용 흐름은 다음과 같습니다.

1. 프로덕션에서 수집한 실제 대화 히스토리를 업로드합니다.
2. 시뮬레이션을 실행해 새 모델의 응답을 기존 모델과 비교합니다.
3. 안전 문제나 품질 저하가 발생하는 케이스를 사전에 식별합니다.

표준 벤치마크가 놓치는 엣지 케이스(edge case)를 실제 트래픽 패턴으로 잡는다는 점이 차별점입니다.

프로덕션 AI 시스템을 운영하는 팀이라면, 모델 업데이트마다 이 단계를 릴리스 체크리스트에 넣는 것이 실용적입니다. 강의 맥락에서는 "모델을 바꾸면 무엇이 달라지는가"를 실데이터로 보여주는 시연 소재로 쓸 수 있습니다.

## 4. Qwen3.6-27B 로컬 오프라인 코딩 에이전트 설정

llama.cpp 핵심 기여자 게오르기 게르가노프(Georgi Gerganov)가 일상 유지보수 작업에 **Qwen3.6-27B** 모델을 사용하는 방법을 공유했습니다. [출처](https://simonwillison.net/2026/Jun/16/georgi-gerganov)

하드웨어는 M2 Ultra 또는 RTX 5090이지만, 핵심은 설정의 단순함입니다.

```
pi -nc --offline
```

`pi` 에이전트에서 네트워크 접속과 클라우드 호출을 제거한 최소 구성입니다. 여기에 자신의 코딩 스타일을 기술한 시스템 프롬프트를 추가합니다.

실전 팁: 시스템 프롬프트에 "이 프로젝트의 코딩 컨벤션, 주석 언어, 변수 명명 규칙"을 3~5줄로 명시하면 일관성이 높아집니다. 클라우드 레이턴시(latency)나 데이터 보안이 문제인 환경에서 로컬 모델 에이전트는 현실적 대안입니다. 27B 규모는 일상적인 코드 수정·리뷰 작업에서 실용적 수준입니다.

## 5. "fix this code"가 수출 규제를 촉발한 사건의 실무 교훈

보안 연구자 케이트 무수리스(Kate Moussouris)에 따르면, Anthropic의 Claude 5 계열 모델이 미국 수출 규제 대상이 된 계기는 정교한 해킹 프롬프트가 아니었습니다. "fix this code"라는 평범한 지시를 취약한 코드에 적용한 것이 트리거였습니다. [출처](https://news.hada.io/topic?id=30565)

모델이 취약점 탐지→수정→테스트 루프를 실행한 것 자체가 문제로 분류됐습니다. 이는 방어적 보안 작업에서 개발자가 매일 쓰는 워크플로와 동일합니다.

실전 팁 두 가지입니다.

첫째, AI 코딩 도구로 기존 코드를 수정할 때 인증·암호화·시스템 접근 관련 코드임을 명시적으로 밝히십시오. "이 함수는 인증 로직입니다. 안전한 방향으로 수정해 주세요"처럼 맥락을 먼저 제공하는 것이 좋습니다.

둘째, 규제 환경이 있는 조직이라면 AI 코딩 도구 사용 정책에 보안 민감 코드 처리 기준을 별도로 정의할 필요가 있습니다.

오늘 신호를 한 줄로 묶으면, AI 코딩 도구는 워크플로 구조화(워크트리 병렬화, 오프라인 에이전트)와 리스크 인식(규제 트리거, 배포 전 검증) 두 축에서 동시에 진화하고 있습니다. 기능이 강력해질수록 사용 구조를 명확히 설계하는 쪽이 실속입니다.
