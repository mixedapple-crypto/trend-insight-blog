## 1. GPT-5.6 패밀리: Luna·Terra·Sol 선택 기준과 비용 구조

2026년 7월 9일, OpenAI가 GPT-5.6 패밀리를 정식 공개했습니다.

세 모델의 입력·출력 토큰 단가(백만 토큰 기준)는 다음과 같습니다.

- **Luna**: $1–$6 (고빈도·저비용 작업)
- **Terra**: 중간 단계 (범용 균형)
- **Sol**: $5–$30 (복잡한 추론·플래그십)

문맥 창은 세 모델 모두 100만 토큰이며, 지식 컷오프는 2026년 2월입니다.

"에이전트 라스트 엑잼(Agents' Last Exam)"—장기 전문 워크플로 벤치마크—에서 Sol이 53.6점을 기록했습니다. Claude Fable 5의 40.5점을 앞선 수치입니다.

단, SWE-벤치 프로(SWE-Bench Pro)에서는 64.6%로 Fable 5의 80%에 못 미쳤습니다. OpenAI는 해당 벤치마크 문항의 약 30%가 오류라고 문제를 제기했습니다.

실전 팁은 이렇습니다. 단가만 보지 말고 추론 토큰 소비량을 함께 측정하십시오. 작업 유형별로 Luna → Terra → Sol 순서로 테스트해 최적 모델을 고정한 뒤 운영 비용을 산출하는 방식이 유효합니다. [출처](https://simonwillison.net/2026/Jul/9/gpt-5-6)

강의 설계에서도 모델 선택 기준(비용·추론 깊이·지연 허용치)을 실습 단계에 포함하면 수강생의 실무 판단력을 키울 수 있습니다.

## 2. ChatGPT Work: 클라우드·데스크톱 환경 분리와 활용법

ChatGPT Work는 앱·파일·브라우저를 넘나들며 다단계 작업을 자율 실행하는 에이전트입니다.

구조상 두 환경이 명확히 나뉩니다.

- **클라우드 Work**: 온라인 실행, 데스크톱과 동기화 안 됨
- **데스크톱 Work**: 로컬 파일·앱 접근 가능, 권한 허용 필요

민감한 로컬 파일을 다루는 개발자라면 데스크톱 환경에서만 처리하도록 워크플로를 설계해야 합니다. 클라우드로 올리면 데이터 처리 경로가 달라집니다.

사용법은 단순합니다. 상위 목표를 입력하면 에이전트가 단계를 분해해 파일 생성·앱 조작·결과 전달을 순차 처리합니다. 마이그레이션 스크립트 검토, 문서 초안 배포, 강의 자료 정리처럼 플랫폼을 횡단하는 반복 작업에 바로 적용할 수 있습니다. [출처](https://openai.com/index/chatgpt-for-your-most-ambitious-work)

환경 분리 구조를 이해하고 쓰는 것과 모르고 쓰는 것은 데이터 처리 정책상 차이가 큽니다.

## 3. Muse Spark 1.1 API: LLM CLI로 즉시 연결하는 법

Meta의 첫 API 공개 스파크(Spark) 모델인 **Muse Spark 1.1**이 에이전트 도구 호출과 컴퓨터 사용 기능을 강화한 상태로 접근 가능해졌습니다.

LLM CLI 생태계를 이미 사용 중이라면 아래 세 단계로 연결할 수 있습니다.

```bash
uv tool install llm
llm install llm-meta-ai
llm keys set meta-ai       # API 키 입력
llm -m meta-ai/muse-spark-1.1 "질문 내용"
```

또는 pip 방식으로도 설치됩니다.

```bash
pip install llm-meta-ai
llm prompt "질문 내용" -m muse-spark-1.1
```

별도 인증 컨텍스트 전환 없이 기존 LLM CLI 워크플로에 Meta 모델을 바로 붙일 수 있다는 점이 실속입니다. [출처](https://simonwillison.net/2026/Jul/9/llm-meta-ai)

강의 실습에서 여러 모델을 동일 CLI로 비교 테스트하는 시나리오에 즉시 활용 가능합니다.

## 4. 에이전트 장기 기억 설계: 일화·의미·절차적 기억 분리 전략

에이전트(Agent)가 긴 작업을 안정적으로 수행하려면 외부 장기 기억 시스템이 필요합니다. 세 가지 저장 유형으로 구분하는 설계가 실용적입니다.

- **일화적(Episodic) 기억**: 과거 의사결정·결과를 시간순으로 저장, 워크플로 재구성에 사용
- **의미적(Semantic) 기억**: 사용자 선호·도메인 지식 등 구조화 데이터, 벡터 임베딩으로 즉시 검색
- **절차적(Procedural) 기억**: 재사용 가능한 프로세스·기법, 매번 재구성하지 않고 호출

실전 설계 원칙은 다음과 같습니다. 에이전트의 활성 컨텍스트 창(단기)과 영구 저장소(장기)를 분리하고, 검색 증강 생성(RAG)으로 필요한 정보만 호출하십시오.

특히 절차적 기억에 에이전트가 무제한 쓰기 권한을 갖지 않도록 설정하는 것이 중요합니다. 무분별한 쓰기는 버그나 의도치 않은 동작 변화로 이어집니다. [출처](https://www.itworld.co.kr/article/4194247/%ec%9d%bc%ed%99%94%c2%b7%ec%9d%98%eb%af%b8%c2%b7%ec%a0%88%ec%b0%a8%ec%a0%81-%ea%b8%b0%ec%96%b5%ec%9c%bc%eb%a1%9c-%ec%99%84%ec%84%b1%ed%95%98%eb%8a%94-ai-%ec%97%90%ec%9d%b4%ec%a0%84%ed%8a%b8-%ec%9e%a5.html)

에이전트 강의에서 이 세 유형을 다이어그램으로 시각화하면 맥락 창 한계를 직관적으로 설명할 수 있습니다.

## 5. AI 코드 검증·LLM 번아웃: 속도보다 구조가 먼저입니다

오늘 실무 현장 신호 두 개가 같은 방향을 가리킵니다.

Atlassian의 2026년 조사에 따르면 개발자의 50%가 AI 출력 품질 불일치를 보고하고, 96%가 AI 출력을 온전히 신뢰하지 않습니다. 그러나 커밋 전에 일관되게 검증하는 비율은 48%에 그칩니다. 실제 개발 시간에서 코드 작성이 차지하는 비중은 16%뿐으로, 나머지는 검증·테스트·배포·인프라에 쏟립니다. [출처](https://www.itworld.co.kr/article/4194723/ai-%ec%83%9d%ec%84%b1-%ec%bd%94%eb%93%9c-%ec%a0%95%eb%a7%90-%eb%af%bf%ea%b3%a0-%eb%b0%b0%ed%8f%ac%ed%95%a0-%ec%88%98-%ec%9e%88%ec%9d%84%ea%b9%8c-%ea%b2%80%ec%a6%9d%c2%b7%eb%b3%b4%ec%95%88%c2%b7.html)

한편 Claude Code·Copilot·Gemini를 매일 쓰는 개발자들 사이에서 "LLM 번아웃" 현상도 관찰됩니다. 직접 설계·구현하던 방식이 프롬프트 작성→검토→수정 반복으로 바뀌면서 인지 피로가 누적되는 패턴입니다. [출처](https://news.hada.io/topic?id=31286)

두 신호가 시사하는 구조는 동일합니다. AI가 코드 생성 속도를 높여도, 검증 구조가 없으면 속도는 오히려 부채로 쌓입니다.

실전 처방은 세 가지입니다. 첫째, 커밋 전 자동 검증 레이어를 워크플로에 고정합니다. 둘째, 로컬 코딩과 리모트 빌드·배포를 분리해 인프라 마찰을 줄입니다. 셋째, AI를 쓸 영역과 직접 작성할 영역을 사전에 구분해 검토 부담을 분산합니다.

오늘 공개된 도구들—GPT-5.6의 비용 계층, ChatGPT Work의 환경 분리, Muse Spark의 CLI 통합—은 모두 "모델 선택"이 아니라 **워크플로 설계**가 핵심임을 가리키고 있습니다.
