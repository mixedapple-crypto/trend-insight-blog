## 1. Anthropic Claude Code: 팀 내 65% PR 병합률과 에이전트 스킬 공개

Anthropic이 2026년 7월 21일 에이전트 스킬(Agent Skills)을 공개했습니다.

에이전트 스킬은 Claude가 외부 도구·API와 연결되는 모듈 단위 역량입니다.

같은 날 Claude Code 팀의 Cat과 Thariq가 밝힌 내부 지표도 주목할 만합니다.

팀이 자체 Slack 통합 도구인 **Claude Tag**를 제품 엔지니어링 업무에 적용한 결과, PR(Pull Request) 병합률이 **65%**에 달했습니다.

사용 방식은 이렇습니다.

팀 Slack 채널에 Claude Tag를 연결하면, 코드 변경 사항을 리뷰하고 PR 토론에 참여합니다.

Anthropic은 내부에서 먼저 기능을 검증한 뒤 외부 공개합니다.

공개 채널에서 사용해 엣지 케이스를 조기에 파악하는 것이 핵심 운영 원칙입니다.

강의·개발 현장에서는 이 수치를 AI 코딩 워크플로 도입 설득 근거로 활용할 수 있습니다.

팀 단위로 Slack 기반 코드 리뷰 자동화를 설계할 때 참조 지표로 쓰기 적합합니다. [출처](https://simonwillison.net/2026/Jul/21/cat-and-thariq)

## 2. Nativ: Mac 로컬 AI 실행 환경 구축법

**Nativ**는 Prince Canuma가 개발한 macOS 데스크톱 앱으로, 2026년 7월 21일 공개됐습니다.

내부적으로 **MLX** 라이브러리를 감싸 Apple Silicon Mac에서 비전·언어 모델을 로컬 실행합니다.

제공 기능은 두 가지입니다.

채팅 인터페이스와 `localhost` API 서버를 함께 제공합니다.

Hugging Face 디렉터리에 이미 캐시된 MLX 모델을 자동 감지하므로 별도 설정 없이 바로 사용 가능합니다.

실전 팁은 세 가지입니다.

첫째, 클라우드 API 비용 없이 프롬프트 반복 실험이 가능합니다.

둘째, `localhost` API 서버를 활성화하면 기존 OpenAI 호환 코드에서 엔드포인트만 교체해 오프라인 앱을 빠르게 빌드할 수 있습니다.

셋째, Apple Neural Engine에 최적화된 MLX 특성상 비전-LLM(Vision-LLM) 프로토타입 속도가 유리합니다.

강의 환경에서는 인터넷 없는 실습실이나 보안 정책이 엄격한 기업 교육 현장에 특히 적합합니다. [출처](https://simonwillison.net/2026/Jul/21/nativ)

## 3. Gemini Flash 3종 분화: 속도·보안·범용 각각의 쓸 자리

Google이 2026년 7월 Gemini Flash 계열 세 모델을 공개했습니다. [출처](https://www.itworld.co.kr/article/4199751/)

모델별 포지션은 다음과 같습니다.

**Gemini 3.6 Flash**는 코딩과 차트 분석 등 멀티모달 작업에 강점을 보입니다.

**Gemini 3.5 Flash Lite**는 저지연 에이전트 검색에 최적화돼 있습니다.

**Gemini 3.5 Flash Cyber**는 사이버 보안 특화 모델로, 현재 정부·파트너사로 접근이 제한됩니다.

3.6과 3.5 Lite는 Gemini 앱에서 지금 바로 사용할 수 있습니다.

문서 요약, 주간 계획 정리 등 추론 깊이보다 속도가 중요한 작업에 적합합니다.

한 가지 구조적 대비가 있습니다.

프로 등급 모델은 여전히 2월 공개된 Gemini 3.1 Pro입니다.

같은 기간 경쟁사들은 Claude Opus 4.8(5월), GPT-5.5(4월), GPT-5.6 Sol(6월 말)을 연달아 공개했습니다.

복잡한 코딩이나 긴 맥락 작업이 필요한 개발자라면 Gemini 3.5 Pro 공개 전까지 프론티어 모델 선택지를 함께 검토할 필요가 있습니다.

## 4. Gemini API 샘플링 파라미터 폐기: 코드 수정 가이드

Gemini 최신 모델은 `temperature`, `top_p`, `top_k` 파라미터를 더 이상 지원하지 않습니다. [출처](https://ai.google.dev/gemini-api/docs/latest-model)

API가 이 값을 조용히 무시하며 오류나 경고를 반환하지 않습니다.

즉, 기존 코드가 정상 작동하는 것처럼 보여도 파라미터가 실제로는 적용되지 않습니다.

실전 대응 방법은 두 가지입니다.

첫째, 출력 다양성·무작위성 제어가 필요하다면 시스템 프롬프트나 프롬프트 엔지니어링으로 대체해야 합니다.

둘째, 기존 Gemini 연동 코드에서 해당 파라미터 사용 여부를 즉시 점검하고 제거하는 것이 좋습니다.

강의 자료에 Gemini API 샘플링 예제가 포함된 경우, 이번 변경으로 실습 결과가 예상과 달라질 수 있습니다.

## 5. AI 코딩 에이전트 샌드박스 취약점: 네 가지 공격 패턴과 대응

Pillar Security가 **Cursor**, Codex, Gemini CLI, Antigravity 등 여러 AI 코딩 에이전트에서 샌드박스 우회 취약점을 보고했습니다. [출처](https://www.itworld.co.kr/article/4199698/)

핵심 구조는 이렇습니다.

에이전트가 파일을 생성하면, 신뢰받는 호스트 컴포넌트가 이를 샌드박스 외부에서 실행합니다.

직접 탈출 없이 격리를 우회하는 방식입니다.

네 가지 취약 패턴은 다음과 같습니다.

첫째, OS 복잡성에서 비롯된 차단 목록(denylist) 샌드박스 허점입니다.

둘째, 실행 가능한 워크스페이스 설정 파일입니다.

셋째, 이름만 신뢰하고 호출 방식을 검증하지 않는 명령 허용 목록(allowlist)입니다.

넷째, Docker Desktop처럼 샌드박스 밖에 있는 권한 있는 로컬 데몬입니다.

실무 대응 팁은 세 가지입니다.

민감한 워크스페이스 설정과 에이전트 생성 자동화 파일은 실행 전 명시적 승인 절차를 두어야 합니다.

파일 출처(사용자 생성 vs 에이전트 생성)를 구분해 관리해야 합니다.

git hook, 가상 환경, 설정 파일 등 파일 기반 공격 면을 별도로 모니터링해야 합니다.

"규칙은 지켰지만 경계는 넘었다"는 이번 보고의 핵심입니다.

샌드박스 존재 자체를 보안의 완성으로 보는 시각이 바뀌어야 할 시점입니다.

오늘 신호 전체를 묶으면, 에이전트 도구가 생산성 영역(Claude Tag, 에이전트 스킬)과 보안 영역(샌드박스 취약점) 모두에서 동시에 성숙하고 있다는 흐름입니다.

로컬 실행(Nativ)과 API 변경(Gemini 파라미터 폐기)은 그 기반 인프라가 조용히 재편되고 있음을 보여 줍니다.
