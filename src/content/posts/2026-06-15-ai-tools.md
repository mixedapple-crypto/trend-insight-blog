## 1. GitHub Copilot CLI 3종 업데이트: 위임 선택성·언어 서버·커스텀 에이전트

**GitHub Copilot CLI**가 이번 주 세 가지 기능을 연달아 발표했습니다.

첫째, 에이전트 위임 선택성 개선입니다. 이전에는 작업 중간에 다른 도구로 넘기거나 사용자 확인을 요청하는 빈도가 높았습니다. 이번 업데이트로 CLI가 독립적으로 완료할 수 있는 범위를 먼저 처리한 뒤 위임을 결정합니다. 별도 설정 변경 없이 적용됩니다. [출처](https://github.blog/ai-and-ml/how-we-made-github-copilot-cli-more-selective-about-delegation)

둘째, 언어 서버 프로토콜(LSP, Language Server Protocol) 통합입니다. 기존 grep 방식 대신 LSP 서버를 연결해 심벌 해석·타입 정보·파일 간 참조를 정확하게 처리합니다.

설정은 Copilot CLI 설정 파일에 LSP 서버 경로를 추가하는 방식입니다. Python이라면 `pylsp`, JavaScript라면 `typescript-language-server`를 지정합니다. 대형 리포지터리에서 쿼리 정확도가 크게 올라갑니다. [출처](https://github.blog/ai-and-ml/github-copilot/give-github-copilot-cli-real-code-intelligence-with-language-servers)

셋째, 커스텀 에이전트(Custom Agent) 워크플로입니다. 팀 스택과 코딩 규칙을 에이전트 정의 파일에 한 번 작성하면, 이후 CLI 호출 시마다 컨텍스트를 재설명할 필요가 없습니다. 에이전트는 도구·지식·제약 조건을 명시적으로 포함하므로 팀 표준 준수 여부를 감사(Audit)할 수 있습니다. [출처](https://github.blog/ai-and-ml/github-copilot/from-one-off-prompts-to-workflows-how-to-use-custom-agents-in-github-copilot-cli)

실전 팁: LSP 통합과 커스텀 에이전트를 함께 쓰면, 에이전트가 정확한 타입 정보를 바탕으로 반복 리팩터링을 실행할 수 있습니다. 강의 환경이라면 커스텀 에이전트에 수강생 코드 컨벤션을 심어 두고, 실습 피드백 자동화에 활용할 수 있습니다.

## 2. Google Colab CLI 공개: 터미널에서 원격 GPU 직접 연결

**Google**이 Colab CLI를 공개했습니다. 로컬 터미널을 원격 Colab 런타임에 연결해 브라우저 전환 없이 GPU 작업을 실행할 수 있습니다. [출처](https://developers.googleblog.com/introducing-the-google-colab-cli)

사용 흐름은 단순합니다. CLI를 설치한 뒤 `colab connect`로 런타임에 접속하면, 로컬 스크립트를 원격 GPU 위에서 실행하고 로그와 결과물(예: 파인튜닝된 모델 가중치)을 로컬로 가져올 수 있습니다.

실전 팁: CI/CD 파이프라인에서 GPU 단계를 Colab에 위임하는 구조를 만들 수 있습니다. **Claude Code** 같은 에이전트 도구와 연결하면, 에이전트가 코드를 생성하고 Colab CLI로 실행 결과를 받아 다음 단계를 결정하는 루프를 구성할 수 있습니다. 로컬 개발 환경에서 GPU가 필요한 실험을 빠르게 분리하려는 강사에게도 유용합니다.

## 3. OpenAI, Ona 인수로 Codex에 영구 클라우드 환경 추가

**OpenAI**가 Ona를 인수한다고 발표했습니다. 목적은 **Codex**에 세션 간 상태를 유지하는 영구 클라우드 실행 환경을 통합하는 것입니다. [출처](https://openai.com/index/openai-to-acquire-ona)

기존 Codex는 단일 요청 단위로 코드를 생성했습니다. Ona 통합 이후에는 에이전트가 이전 세션의 컨텍스트와 파일 상태를 유지한 채 장기 작업을 이어 실행할 수 있게 됩니다. 기업 워크플로에서 반복 작업을 에이전트에 위임하는 패턴이 실용 범위 안으로 들어옵니다.

현재 공개된 통합 일정은 없습니다. 단, 이번 인수는 OpenAI가 단발성 코드 생성이 아닌 상태 유지 에이전트 방향으로 Codex를 전환하고 있음을 명확히 보여줍니다.

## 4. DiffusionGemma: 순차 처리 없는 확산 기반 추론 모델

**Google**이 270억 매개변수(Parameter) 오픈소스 모델 **DiffusionGemma**를 공개했습니다. 기존 자기회귀(Autoregressive) 방식과 달리, 플레이스홀더 토큰에서 시작해 여러 번의 정제(Refinement) 과정으로 전체 텍스트 블록을 동시에 생성합니다. [출처](https://www.itworld.co.kr/article/4184835/%ea%b5%ac%ea%b8%80-%ec%88%9c%ec%b0%a8-%ec%b2%98%eb%a6%ac-%ed%83%88%ed%94%bc%ed%95%9c-%ed%99%95%ec%82%b0-%ea%b8%b0%eb%b0%98-ai-%eb%aa%a8%eb%8d%b8-%eb%94%94%ed%93%a8%ec%a0%84%ec%a0%ac%eb%a7%88.html)

추론 시 실제 활성화되는 매개변수는 39억 개입니다. RTX 5090 기준 양자화(Quantization) 적용 시 VRAM 18GB로 구동됩니다. GPU 단일 사용자 환경에서 최대 4배 빠른 추론 속도를 보고합니다.

실전 팁: 인라인 편집이나 비순차 텍스트 작업(예: 중간 단락 수정, 코드 블록 내부 삽입)에서 양방향 어텐션(Bidirectional Attention)의 이점이 드러납니다. Apache 2.0 라이선스로 Hugging Face, Google Cloud Model Garden, NVIDIA NIM에서 내려받을 수 있습니다. 토큰 단위 과금 구조를 벗어나 비용 예측이 가능한 로컬 추론 환경을 원하는 팀에 적합합니다.

## 5. AI가 소프트웨어 엔지니어를 대체하지 못하는 구조적 이유

Narayanan·Kapoor의 분석과 뉴욕주 AI 공시 의무화 데이터(160개 이상 기업, 2025년)가 같은 결론을 가리킵니다. AI 관련 해고를 나타내는 WARN 공지는 0건이었습니다. [출처](https://simonwillison.net/2026/Jun/14/why-ai-hasnt-replaced-software-engineers)

병목은 코딩 속도가 아닙니다. 무엇을 만들지 결정하는 판단, 결과물에 대한 책임, 코드베이스와 비즈니스 도메인의 깊은 맥락 이해가 자동화되지 않는 구간입니다.

AI가 코딩 속도를 높일수록, 그 위에서 작동하는 의사결정·검증·도메인 전문성의 희소성은 오히려 커집니다. 오늘 소개한 Copilot CLI 커스텀 에이전트나 Codex 상태 유지 환경도 같은 구조 안에 있습니다. 도구가 실행을 맡을수록, 실무자의 역할은 '어디로 실행할지'를 정의하는 쪽으로 이동합니다. 도구 흐름 전체가 이 방향을 가리키고 있습니다.
