## 1. Kimi K3: OpenRouter로 바로 쓰는 법과 가격 구조

Moonshot AI가 2026년 7월 16일 **Kimi K3**를 API로 공개했습니다. 총 2.8조 파라미터 중 500억을 활성화하는 혼합 전문가(MoE) 구조이며, 7월 27일까지 오픈 웨이트 공개도 예고된 상태입니다.

가격은 입력 100만 토큰당 3달러, 출력 100만 토큰당 15달러입니다. 직전 모델 K2.6 대비 출력 토큰이 21% 줄어 실질 비용 부담은 더 낮아집니다.

Arena.ai 프런트엔드 코드 벤치마크 1위, 장기 지식 작업 엘로(Elo) 1547을 기록했습니다. 일부 평가에서는 Claude Fable 5·GPT-5.6 Sol에 뒤지므로, 코딩 중심 작업에 특히 유리합니다.

**실전 사용법 — OpenRouter 경유:**

```bash
# llm-openrouter 플러그인 설치
pip install llm-openrouter

# 모델 호출 (Moonshot 키 불필요)
llm -m openrouter/moonshotai/kimi-k3 '여기에 프롬프트 입력'
```

명령 실행 후 토큰 사용량과 비용 명세가 자동으로 출력됩니다.

강의나 코드 리뷰 워크플로에 빠르게 끼워 넣고 Claude Sonnet과 출력 품질·비용을 나란히 비교해 보는 것이 가장 빠른 검증 방법입니다. [출처](https://simonwillison.net/2026/Jul/16/kimi-k3)

## 2. Inkling: Apache 2.0 멀티모달 오픈 모델 실전 활용

Thinking Machines Lab이 **Inkling**을 공개했습니다. 총 9750억 파라미터(활성 410억), 45조 토큰으로 학습한 멀티모달 모델이며 텍스트·이미지·오디오·영상을 단일 아키텍처에서 처리합니다.

라이선스는 **Apache 2.0**으로 상업·연구 용도 모두 제약이 없습니다. 2760억 파라미터 경량 버전 Inkling-Small도 함께 공개됐습니다.

모델 설계 목표는 프런티어 성능보다 **파인튜닝 기반**입니다. 자체 Tinker 플랫폼에서 도메인 특화 학습을 지원하며, Ollama·vLLM 등 표준 프레임워크로 바로 로드할 수 있습니다.

실전 팁 세 가지입니다.

- 벤더 종속 없이 사내 추론 서버를 구축할 때 기본 선택지로 검토하십시오.
- Tinker 플랫폼으로 사내 문서·코드베이스를 학습시켜 특화 어시스턴트를 만들 수 있습니다.
- 모델 카드 분량이 적으므로, 세부 학습 데이터 구성은 추가 문서를 기다린 뒤 프로덕션 투입 여부를 결정하십시오.

강의용 데모나 기업 내부 도구에 Apache 2.0 모델이 필요한 팀에게 현시점 가장 현실적인 대형 오픈 모델 옵션입니다. [출처](https://simonwillison.net/2026/Jul/16/inkling)

## 3. Codex 전체 접근 모드의 홈 디렉터리 삭제 버그 대응

GPT-5.6의 코덱스(Codex) 기능에서 위험한 버그가 보고됐습니다. 전체 접근(Full Access) 모드에서 샌드박스 없이 실행하면, 모델이 `$HOME`을 임시 디렉터리로 잘못 덮어쓴 뒤 사용자 홈 전체를 삭제할 수 있습니다.

복구가 어려운 데이터 소멸 사고이므로 즉시 적용할 수 있는 대응이 필요합니다.

**필수 체크리스트:**

1. 신뢰하지 않는 코드에는 전체 접근 모드를 비활성화합니다.
2. 샌드박스 보호 옵션을 활성화합니다.
3. Codex 실행 전 자동 검토(Auto-review)를 반드시 켭니다.

에이전트(Agent) 기반 도구를 프로덕션 환경에 연결할 때, 모델의 실수 한 번이 시스템 전체에 영향을 줄 수 있다는 점을 다시 확인시켜 주는 사례입니다. 강의에서 AI 에이전트 보안을 다룰 때 구체적인 실사례로 활용할 수 있습니다. [출처](https://simonwillison.net/2026/Jul/16/bad-codex-bug)

## 4. Firefox-in-WebAssembly: Claude로 가속한 복잡한 포팅 사례

Puter가 Firefox를 웹어셈블리로 컴파일해 다른 브라우저(예: Chrome) 안에서 구동하는 데 성공했습니다. 코드 생성에 **Claude Opus**를 활용했으며, 예상 토큰 비용 2만 5000달러를 Claude Max 구독으로 절감했습니다.

브라우저는 임의의 네트워크 연결을 직접 열 수 없으므로, 모든 트래픽은 Wisp 프로토콜 기반 웹소켓(WebSocket) 프록시를 통해 라우팅합니다. HTTPS 트래픽에는 종단간 암호화를 적용했습니다.

오픈소스 저장소(firefox-wasm)는 공개된 상태입니다.

실전 시사점은 두 가지입니다.

- 전통적인 엔지니어링 단독으로는 일정이 맞지 않는 복잡한 포팅 작업에 LLM을 코드 생성 가속기로 쓰는 패턴을 보여주는 구체적 사례입니다.
- 서버 사이드 프록시 비용과 지연이 있으므로 프로덕션보다는 샌드박스 격리·크로스플랫폼 테스트 목적에 적합합니다.

[출처](https://simonwillison.net/2026/Jul/16/firefox-in-webassembly)

## 5. 구형 Xeon에서 Gemma 4 26B 실행 — 하드웨어 없는 팀을 위한 옵션

**Gemma 4 26B**가 13년 된 듀얼 Xeon E5-2690 v2(DDR3 RAM)에서 초당 약 5토큰, 프롬프트 평가 기준 초당 16토큰으로 동작하는 결과가 보고됐습니다. [출처](https://news.hada.io/topic?id=31498)

llama.cpp의 Q8_0 양자화(Quantization)와 AVX1·스칼라 폴백을 활용했습니다. AVX2·FMA3 명령어 세트 없이도 작동한다는 점이 핵심입니다.

실전 팁입니다.

- GPU 예산이 없는 팀은 유휴 서버나 구형 워크스테이션을 llama.cpp 추론 노드로 전환할 수 있습니다.
- 강의 실습 환경처럼 응답 속도보다 비용이 우선인 상황에 적합합니다.
- Q8_0보다 낮은 양자화 수준(Q4_K_M 등)을 적용하면 속도를 추가로 높일 수 있습니다.

오늘 신호를 한 줄로 묶으면 이렇습니다. 대형 오픈 모델 두 개가 같은 날 등장해 API 비용과 라이선스 두 축에서 선택지가 넓어졌습니다. 반면 Codex 버그는 에이전트 도구의 실수 허용 범위가 여전히 좁다는 점을 상기시킵니다. 새 모델을 빠르게 실험하되 실행 권한 설정을 먼저 점검하는 순서가 안전합니다.
