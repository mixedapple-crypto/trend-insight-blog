## 1. Datasette 1.0a35: GUI 테이블 생성·수정 + JSON API

**Datasette 1.0a35**가 2026년 6월 23일 공개됐습니다.

데이터베이스 액션 메뉴에 "Create table" 화면이 추가됐습니다. 컬럼 정의, 기본 키, 커스텀 타입, 제약 조건, 기본값, 외래 키를 GUI에서 설정할 수 있습니다.

`/<database>/-/create` 엔드포인트를 직접 호출하면 같은 작업을 JSON API로 처리합니다. 마이그레이션 자동화나 어드민 인터페이스 구축에 바로 활용 가능합니다.

**"Alter table"** 기능도 함께 추가됐습니다. `/<database>/<table>/-/alter` API로 컬럼 추가·이름 변경·순서 조정·삭제·제약 조건 수정을 SQL 없이 처리합니다.

Template context(템플릿 컨텍스트) 문서화도 이번 릴리스에 포함됐습니다. 버전 2.0까지 안정적인 API 계약을 제공하므로, Datasette 핵심 페이지를 확장하는 개발자는 하위 호환 걱정을 줄일 수 있습니다.

강의 환경에서는 "수강생이 SQL 없이 스키마를 직접 다루는 실습"에 이 GUI를 바로 활용할 수 있습니다. [출처](https://simonwillison.net/2026/Jun/23/datasette)

## 2. GLM-5.2 로컬 실행: 744B 파라미터, VRAM 24GB부터

Z.ai의 **GLM-5.2**는 총 744B 파라미터에 활성 파라미터 40B, 컨텍스트 창 100만 토큰 규모의 오픈 모델입니다.

Unsloth Dynamic GGUF(지지유에프) 양자화 도구로 4비트 또는 8비트 정밀도로 로드하면, VRAM 24~48GB 소비자용 GPU에서 전체 컨텍스트 추론이 가능합니다.

실전 팁은 세 가지입니다.

첫째, 4비트 양자화는 VRAM 24GB(예: RTX 3090) 단일 GPU에서 동작을 확인했습니다.

둘째, 장기 코딩 태스크와 멀티 턴 추론 성능이 Claude·GPT급과 경쟁하는 수준으로 보고됐습니다.

셋째, API 비용과 지연을 없애야 하는 로컬 코딩 어시스턴트 구축에 GLM-5.2가 현실적인 선택지가 됩니다.

강의 환경에서는 인터넷 없이 동작하는 오프라인 코딩 실습 서버 구성에 바로 적용할 수 있습니다. [출처](https://news.hada.io/topic?id=30760)

## 3. Claude Code 추론 블록: 암호화 서명 구조 이해

**Claude Code**의 "Extended Thinking(확장 추론)" 기능은 추론 출력을 암호화 서명으로 저장합니다.

사용자 기기에는 약 600자 길이의 서명 문자열만 남습니다. 복호화 키는 Anthropic만 보유하므로, 디스크 로그에서 실제 추론 경로를 직접 확인할 수 없습니다.

이 구조는 Anthropic의 지적 재산(IP) 보호를 우선합니다. AI 동작 디버깅이나 모델 로직 검증이 필요한 개발·교육 상황에서는 제약으로 작용합니다.

실전 팁은 두 가지입니다.

첫째, 추론 블록을 '근거 확인' 수단으로 쓰려 했다면 설계 전제를 바꿔야 합니다.

둘째, 모델 판단 과정을 외부에서 검증해야 하는 기관 환경이라면, 추론 로그가 오픈된 대안 모델을 병행 검토하는 것이 현실적입니다. [출처](https://news.hada.io/topic?id=30766)

## 4. AI 에이전트 컨텍스트 유지: 설정 파일 전략

Claude, ChatGPT 같은 AI 코딩 에이전트는 새 세션마다 이전 컨벤션·코드 스타일·버그 해결 방식을 초기화합니다. 이를 보완하는 패턴이 실무 커뮤니티에서 공유됐습니다.

핵심 방법은 프로젝트 루트에 컨텍스트 파일을 두는 것입니다.

- **Claude**: 프로젝트 기능의 "Project instructions"에 팀 컨벤션 저장
- **Cursor**: `.cursor/rules` 파일 또는 Rules 탭에 지시사항 기재
- **범용**: `.gn-agent-context` 파일에 TypeScript `type` 사용 규칙, PR 리뷰 기준, 반복 발생 버그 패턴 문서화

파일에 담을 내용은 구체적일수록 효과가 높습니다. "함수는 단일 책임"보다 "에러 핸들링은 항상 try-catch + 로거 호출 패턴 사용"처럼 코드 수준으로 기술하면 에이전트가 일관되게 따릅니다.

팀 단위에서는 이 파일을 Git으로 버전 관리하면, 신규 팀원이 에이전트를 쓸 때도 동일한 컨텍스트에서 시작합니다. 강의에서는 수강생별 편차를 줄이는 실습 표준화 도구로 활용 가능합니다. [출처](https://news.hada.io/topic?id=30768)

## 5. '에이전틱 AI PC' 마케팅 vs 실속 점검

Nvidia와 Microsoft가 'Agentic AI PC'를 새로운 패러다임으로 내세우고 있습니다. 그러나 분석가들은 이를 NPU 칩이 탑재된 기존 AI PC의 재포장으로 봅니다.

핵심 하드웨어 차별점은 Nvidia의 N1X 칩(Arm 기반 CPU+GPU 통합)입니다. 다만 x86 기반 기업 소프트웨어 호환성 문제와 킬러 앱 부재가 단기 기업 도입의 벽으로 지적됩니다.

실무적으로 확인할 사항은 하나입니다. 현세대 PC에서도 OpenInterpreter 같은 로컬 에이전트는 클라우드 없이 이미 동작합니다. '에이전틱' 브랜드가 새 하드웨어 구매 명분이 되려면 구체적인 기능 차이를 먼저 확인해야 합니다.

오늘 신호의 공통 구조는 '기능 추가와 실체 확인의 대비'입니다. Datasette의 GUI·API 확장은 즉시 쓸 수 있는 실속이고, AI PC 마케팅과 Claude 추론 블록 불투명성은 과장과 설계 의도를 구분해서 읽어야 하는 신호입니다. GLM-5.2 로컬 실행과 에이전트 컨텍스트 파일 전략은 오늘 당장 파이프라인에 적용 가능한 두 가지 실용 경로입니다. [출처](https://www.itworld.co.kr/article/4188631)
