## 1. OpenAI Codex: 부서별 토큰 사용량 56배 성장이 가리키는 것

**OpenAI**는 2025년 11월 이후 내부 **Codex** 사용 지표를 공개했습니다. 중앙값 출력 토큰 기준으로 리서치 부문 56배, 고객지원 32배, 엔지니어링 27배, 법무 13배 성장을 기록했습니다. [출처](https://www.latent.space/p/ainews-openai-reports-median-internal)

이 수치가 의미하는 바는 명확합니다. 짧은 코드 스니펫 생성이 아니라, 긴 컨텍스트를 소비하는 에이전트(Agent) 작업이 일상화되었다는 뜻입니다.

실무 적용 관점에서 주목할 점은 두 가지입니다.

첫째, 출력 토큰이 늘수록 검증 비용도 비례해 증가합니다. 에이전트가 생성한 장문 코드를 그대로 병합하면 리뷰 사각지대가 생깁니다. 체크포인트를 250줄 단위로 끊어 검토하는 습관이 필요합니다.

둘째, 부서별 성장률 차이가 큽니다. 리서치(56배) 대 법무(13배)의 격차는 도메인 특성과 위험 허용 범위가 채택 속도를 결정한다는 점을 보여줍니다. 강의 커리큘럼을 설계할 때 수강생의 도메인별 수용 속도를 구분해 난이도를 조정하는 근거로 쓸 수 있습니다.

## 2. GitHub Copilot 에이전트 하네스: 20개 이상 모델 비교 평가법

**GitHub Copilot** 에이전트 하네스(Agentic Harness)가 20개 이상 모델을 대상으로 벤치마크 데이터를 제공하기 시작했습니다. [출처](https://github.blog/ai-and-ml/github-copilot/evaluating-performance-and-efficiency-of-the-github-copilot-agentic-harness-across-models-and-tasks)

핵심은 성능과 토큰 효율을 같은 작업 단위로 비교할 수 있다는 점입니다. 비용이 높은 모델이 항상 최선이 아님을 수치로 확인할 수 있습니다.

실전 워크플로는 다음 순서를 권장합니다.

1. 대상 태스크(코드 리뷰, 테스트 생성, 문서화 등)를 유형별로 분류합니다.
2. 하네스에서 각 유형에 맞는 모델 3~4개를 동일 프롬프트로 실행합니다.
3. 정확도 대비 소비 토큰 비율을 기록해 태스크별 최적 모델을 고정합니다.

팀 규모가 클수록 모델 선택을 태스크 단위로 분리하면 API 비용을 30~40% 줄이는 사례가 보고됩니다. 강의에서는 이 하네스를 "모델 선택 의사결정 프레임워크" 실습 자료로 바로 활용할 수 있습니다.

## 3. AI 법적 책임: 독일 법원 구글 판결과 개발자 대응 전략

독일 법원이 **Google**의 AI 요약 오류에 대해 회사 책임을 인정했습니다. AI 에이전트를 조직의 대리인으로 간주해, 인간 작성자와 동일한 정확성 기준을 적용한 판결입니다. [출처](https://simonwillison.net/2026/Jun/25/ai-and-liability)

이 판결은 "AI가 생성한 것이니 책임 없다"는 방어 논리를 차단합니다. 배포 조직이 콘텐츠 품질에 대한 법적 책임을 집니다.

개발자와 강사가 지금 당장 적용할 수 있는 체크리스트입니다.

- AI 생성 콘텐츠를 외부에 게시하기 전 인간 검토 단계를 명시적으로 워크플로에 삽입합니다.
- 오류 발생 시 수정 이력을 남길 수 있는 버전 관리 체계를 갖춥니다.
- 클라이언트 납품물에 AI 생성 여부를 명시하고 검증 책임 소재를 계약서에 포함합니다.

한국도 AI 생성물 책임 관련 법제 논의가 진행 중입니다. 지금 검토 절차를 만들어 두는 것이 나중의 리스크 관리 비용을 줄입니다.

## 4. Anthropic vs Alibaba: 모델 증류 공격과 API 보호 실전 팁

**Anthropic**이 **Alibaba**가 Claude 모델 출력을 대규모로 수집해 자사 모델 학습에 활용했다고 주장했습니다. 미국 상원 AI 청문회를 앞두고 제기된 이 주장은 API 접근 가능한 모델의 구조적 취약점을 드러냅니다. [출처](https://news.hada.io/topic?id=30820)

모델 증류(Model Distillation) 공격은 강한 모델의 출력을 반복 수집해 약한 모델을 빠르게 개선하는 방식입니다. 이 패턴은 OpenAI 이용 약관 위반이자 지식재산권 분쟁의 중심으로 부상하고 있습니다.

자체 AI 서비스를 운영하거나 API를 노출하는 팀을 위한 방어 전략입니다.

- 단일 키로 대량 쿼리가 들어오면 자동 차단하는 **속도 제한(Rate Limiting)**을 설정합니다.
- 응답 길이를 태스크에 필요한 최솟값으로 제한해 데이터 추출 효율을 낮춥니다.
- 비정상 쿼리 패턴(동일 유형 반복, 체계적 변형)을 모니터링하는 로그 파이프라인을 구성합니다.
- 프로덕션 모델 출력에 사용 목적 외 재학습 금지 조항을 이용약관에 명시합니다.

## 5. AI 도구 피로감과 실행 거버넌스: 과장과 실속 사이

오늘 신호들은 두 방향의 대비를 보여줍니다.

한쪽에는 Codex 56배 성장, Copilot 멀티모델 하네스처럼 AI 도구 도입 속도를 강조하는 수치가 있습니다.

다른 쪽에는 조용한 경고들이 있습니다. Claude와 Cursor를 매일 쓰는 개발자들은 반복적인 컨텍스트 재설정과 결과 검증으로 인지 피로를 호소합니다. [출처](https://news.hada.io/topic?id=30841) Gartner는 실행 거버넌스(Execution Governance) 부재로 40% 이상의 에이전트 프로젝트가 2027년까지 취소될 것으로 추정합니다. [출처](https://www.itworld.co.kr/article/4189709/%ec%9e%98-%eb%a7%8c%eb%93%a0-ai-%ec%97%90%ec%9d%b4%ec%a0%84%ed%8a%b8%eb%8f%84-%ec%8b%a4%ed%8c%a8%ed%95%98%eb%8a%94-%ec%9d%b4%ec%9c%a0%eb%8b%b5%ec%9d%80-%ec%8b%a4%ed%96%89-%ea%b1%b0.html)

실속 있는 접근은 도구의 확장이 아니라 경계 설정입니다. AI 코딩 세션을 집중 블록으로 묶고, 수락 기준을 명확히 정의하고, 감사 추적(Audit Trail)을 남기는 구조가 장기 생산성을 지탱합니다. 오늘 전체 신호를 한 줄로 압축하면 이렇습니다. 도구의 속도는 빨라졌고, 그 속도를 감당할 거버넌스는 아직 따라오지 못하고 있습니다.
