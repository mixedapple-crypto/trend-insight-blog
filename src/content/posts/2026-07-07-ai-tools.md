## 1. Tencent Hy3: 295B MoE 모델, 7월 21일까지 무료

Tencent가 **Hy3**를 Hugging Face에 공개했습니다. 파라미터 총량은 2,950억 개(295B)이지만, 실제 추론 시 활성화되는 파라미터는 210억 개(21B)에 그치는 혼합전문가(Mixture-of-Experts, MoE) 구조입니다.

컨텍스트 창은 **256K 토큰**으로, 긴 코드베이스 전체를 한 번에 넣거나 장문 문서를 분석하는 작업에 적합합니다. Tencent는 동급 성능 대비 파라미터가 2~5배 많은 모델과 경쟁 가능하다고 주장합니다.

**사용법은 두 갈래입니다.**

로컬 실행이 필요하다면 Hugging Face에서 전체 모델(598GB) 또는 FP8 양자화 버전(300GB)을 내려받습니다. 라이선스는 Apache 2.0으로 상업 활용이 가능합니다.

빠른 테스트가 목적이라면 **OpenRouter**에서 7월 21일까지 무료로 API 호출할 수 있습니다. 로컬 GPU 없이도 256K 컨텍스트를 바로 체험할 수 있습니다.

실전 팁: 코드 생성 시 리포지터리 전체를 컨텍스트에 담아 전달하거나, 계약서·보고서 같은 장문 문서 분석 과제에 먼저 투입해 보십시오. 무료 기간 안에 프로젝트별 성능을 비교해 두면 7월 21일 이후 비용 판단에 유리합니다. [출처](https://simonwillison.net/2026/Jul/6/hy3)

## 2. DRIFT: Mac·CUDA 혼합으로 대형 LLM 레이어 분산 실행

**DRIFT**는 단일 LLM을 여러 기기에 레이어 단위로 나눠 실행하는 오픈소스 프로젝트입니다. Mac과 CUDA GPU를 이기종(heterogeneous) 조합으로 엮어, 각 기기의 여유 용량에 따라 레이어를 배분합니다.

기존 로컬 실행의 병목은 단일 기기의 VRAM 한계였습니다. DRIFT는 이 한계를 여러 기기로 분산해 돌파합니다. 고가의 API 비용 없이 대형 모델을 완전히 로컬에서 제어할 수 있습니다.

실전 팁: 사무실 Mac과 개인 CUDA 워크스테이션을 같은 네트워크에 묶어 구성하면, 비용 부담 없이 Hy3급 모델을 실험할 수 있습니다. 데이터 프라이버시가 중요한 내부 코드베이스 작업이나 오프라인 환경 강의 실습에 특히 유용합니다.

다만 네트워크 지연이 추론 속도에 영향을 줍니다. 기기 간 대역폭을 먼저 측정한 뒤 레이어 배분 비율을 조정하는 것이 권장됩니다. [출처](https://news.hada.io/topic?id=31177)

## 3. Ollama 로컬 워크벤치: 개발 워크플로에 끼워 넣기

Ollama는 오픈소스 모델을 명령줄 몇 줄로 내려받고 실행합니다. GPU 환경 설정의 진입 장벽을 낮춘 것이 핵심입니다.

활용 포인트는 '대체'가 아닌 '보완'입니다. 복잡한 추론이나 최신 기능이 필요할 때는 클라우드 모델을 쓰고, 코드 자동 완성·문서 초안·비공개 코드베이스 테스트처럼 반복 빈도가 높고 프라이버시가 중요한 작업에는 로컬 모델을 끼워 넣는 구조입니다.

실전 팁: `ollama run <모델명>` 한 줄로 실행 후, VS Code 확장(Continue 등)과 연결하면 인터넷 없이 코드 어시스턴트를 구성할 수 있습니다.

강의 환경에서는 수강생 각자의 노트북에 Ollama를 설치해 두면, API 키 없이도 LLM 실습을 진행할 수 있습니다. 입문 강의의 셋업 장벽을 크게 줄여 줍니다. [출처](https://yozm.wishket.com/magazine/detail/3835)

## 4. Claude 구독 정책 논란: 서드파티 연동 비용 구조 변화

Anthropic이 Claude 구독 적용 범위를 자사 도구(Claude Code CLI·Desktop, Claude CoWork, Slack의 @Claude)로 한정하고, 서드파티 에이전트(Agent) 연동에는 별도 과금을 적용하는 방향을 추진 중입니다.

구독료를 냈더라도 외부 오케스트레이션 레이어나 비공식 클라이언트를 통해 Claude를 호출하면 추가 비용이 발생합니다. 이는 OpenAI가 구독 티어에서 더 넓은 연동 경로를 허용하는 방식과 대비됩니다.

실무 판단 기준: 팀이 Claude를 비네이티브 플랫폼에서 주로 사용한다면 총비용을 다시 계산해야 합니다. 네이티브 도구 중심으로 워크플로를 재편하거나, 대안 LLM 제공자와 비용을 비교하는 시점입니다. [출처](https://news.hada.io/topic?id=31201)

## 5. AI 책임 체계와 오늘의 인프라 흐름

SHI International 등 엔터프라이즈 AI 리더들이 공통으로 강조하는 원칙은 하나입니다. **AI 책임은 문서가 아닌 운영에 새겨야 합니다.**

실질적 진단 질문이 있습니다. "AI 시스템이 내일 유해한 응답을 생성하면, 사후 보고서를 누가 작성하는가?" 즉각 답이 나오지 않는다면 책임 구조가 아직 운영에 내재화되지 않은 것입니다.

구체적 조치는 데이터 분류, 접근 통제, 감사 추적, 에스컬레이션 채널을 배포 전에 워크플로에 직접 편입하는 것입니다. 배포를 확대한 뒤에 거버넌스를 얹으면 문제 발생 시 구조가 무너집니다. [출처](https://www.itworld.co.kr/article/4193507/%eb%ac%b8%ec%84%9c-%ec%95%84%eb%8b%8c-%ec%9a%b4%ec%98%81%ec%97%90-%ec%83%88%ea%b2%a8%ec%95%bc-ai-%ec%b1%85%ec%9e%84-%ec%b2%b4%ea%b3%84%eb%a5%bc-%ed%99%95%eb%b3%b4%ed%95%98%eb%8a%94-6%ea%b0%80.html)

오늘의 흐름을 하나로 묶으면, '모델을 어디서 돌리느냐'는 선택지는 넓어졌고 '누가 책임지느냐'는 질문은 더 구체적으로 답해야 하는 시점입니다. Hy3 무료 기간·DRIFT 분산 실행·Ollama 로컬 워크벤치는 각각 비용·하드웨어·프라이버시라는 서로 다른 문제를 겨냥한 해법이며, Claude 구독 논란은 그 선택의 비용 계산을 다시 하게 만드는 변수입니다.
